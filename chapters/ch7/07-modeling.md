# 7.7 设计图而非算法——建模的艺术

> **问题情境**：Skiena的忠告："设计图，而非算法"。问题的建模比算法实现更重要。LLM可以帮你写BFS，但无法帮你判断"这个问题应该建模为图吗"。

---

## 问题情境：建模错误代价更高

小明在某咨询公司实习，接到一个任务：优化会议排期系统。

公司有10个会议室，每天有30场会议，每场会议需要特定会议室和时间。目标是最大化会议数量。

小明花了3天实现了一个复杂的贪心算法，但客户试用后发现：某些会议无法排期，因为会议室冲突。

问题根源：小明建模错了！

**错误建模**：
- 小明把"会议室"建模为节点，试图找最优匹配
- 但忽略了"时间"约束

**正确建模**：
- 会议是节点，时间+会议室是边
- 这是一个流网络问题（最大流=最大会议数）

小明优化了错误的问题。所有算法改进都是浪费。

**这一节要回答的核心问题**：**如何将现实问题建模为图问题？建模错误的代价是什么？LLM能替代建模吗？**

---

## 直观思路：问题→图→算法

### 建模的三步

**第一步：识别图结构**

现实问题有什么"节点"和"边"？

| 现实问题 | 节点 | 边 |
|---------|------|----|
| 课程排课 | 课程 | 先修依赖 |
| 社交网络 | 用户 | 好友关系 |
| 地图导航 | 地点 | 道路连接 |
| 会议排期 | 会议+时间+会议室 | 可用性+冲突 |
| 任务调度 | 任务 | 依赖关系 |

**第二步：选择图类型**

- 有向还是无向？
- 加权还是无权？
- 稀疏还是稠密？

**第三步：匹配标准算法**

| 问题类型 | 图类型 | 标准算法 |
|---------|--------|---------|
| 任务顺序 | DAG | 拓扑排序 |
| 最短路径 | 加权图 | Dijkstra/Bellman-Ford |
| 最小连通 | 无向加权图 | MST |
| 资源分配 | 流网络 | 最大流 |
| 匹配问题 | 二部图 | 最大匹配/最大流 |

### 关键洞察

**建模比算法更重要**：

- 算法错误：找bug、换算法、调参数——有明确改进路径
- 建模错误：解错了问题——所有优化都是浪费，需要从头开始

**建模错误代价更高**：

- 建模错误没有"错误提示"，算法在错误问题上表现完美
- 现实问题有隐式约束，建模时容易遗漏

## 规范定义：建模框架

### 建模五问法

**Skiena六问法**的图建模简化版：

1. **节点是什么？**
   - 问题中的实体是什么？
   - 每个实体是一个节点

2. **边是什么？**
   - 实体之间有什么关系？
   - 关系对应边（有向/无向）

3. **权重是什么？**
   - 关系的强度或成本？
   - 权重对应边权

4. **图类型是什么？**
   - 有向？无向？加权？
   - 稀疏？稠密？

5. **标准算法是什么？**
   - 问题目标对应什么图问题？
   - 选择哪个标准算法？

### 常见建模模式

| 现实问题 | 建模模式 | 图问题 | 标准算法 |
|---------|---------|---------|---------|
| 课程排课 | DAG模式 | 拓扑排序 | Kahn/DFS |
| 地图导航 | 加权图模式 | 最短路径 | Dijkstra/A* |
| 网络设计 | MST模式 | 最小生成树 | Prim/Kruskal |
| 资源分配 | 流网络模式 | 最大流 | Ford-Fulkerson |
| 社交距离 | BFS模式 | 最短距离 | BFS |
| 匹配问题 | 二部图模式 | 最大匹配 | 最大流 |
| 网络分析 | SCC模式 | 连通分量 | Kosaraju |

## 建模案例：从问题到图

### 案例1：课程排课 → DAG → 拓扑排序

**问题**：课程有先修要求，确定选课顺序。

**建模**：
1. 节点：课程
2. 边：先修依赖（有向，A→B表示A先修）
3. 权重：无
4. 图类型：有向图（可能无环）
5. 算法：拓扑排序

**隐式约束**：
- 循环依赖需要检测（环检测）
- 某些课程可能没有先修要求（入度为0）

### 案例2：地图导航 → 加权有向图 → 最短路径

**问题**：从A到B的最快路线。

**建模**：
1. 节点：地点
2. 边：道路连接（有向，双向道路用两条边）
3. 权重：道路长度/时间
4. 图类型：加权有向图
5. 算法：Dijkstra（无负权）或Bellman-Ford（有负权）

**隐式约束**：
- 单行道：单向边
- 送货补贴：负权边（需要Bellman-Ford）

### 案例3：物流配送 → 流网络 → 最大流

**问题**：从仓库到目的地，最大化配送量。

**建模**：
1. 节点：仓库、中转站、目的地
2. 边：运输路线
3. 权重：路线容量
4. 图类型：流网络（有源和汇）
5. 算法：最大流

**隐式约束**：
- 多仓库：多源点
- 多目的地：多汇点

### 案例4：社交网络推荐 → 二部图 → 最大匹配

**问题**：男生和女生配对，最多能配多少对？

**建模**：
1. 节点：男生集合M、女生集合F
2. 边：可配对关系
3. 权重：无
4. 图类型：二部图
5. 算法：最大匹配（或最大流）

**隐式约束**：
- 每人只能配一对：流网络中每人容量为1

### 案例5：会议排期 → 流网络 → 最大流

**问题**：会议+时间+会议室，最大化排期会议数。

**建模**：
1. 节点：会议、时间段、会议室
2. 边：可用性（会议可在某时间某会议室）
3. 权重：容量（会议室在某时间的容量）
4. 图类型：流网络
   - 源点：会议需求
   - 汇点：会议室容量
5. 算法：最大流

**关键洞察**：小明最初把"会议室"建模为节点，忽略了时间约束，导致建模错误。

## 建模错误的代价

### 案例：Psychic Modeling（Skiena War Story）

彩票公司要求：44选6中买15个号码，保证至少4个命中。

Skiena团队用模拟退火找到28张票的"最优"方案，但客户的5张票方案打破了它。

为什么？客户隐式覆盖了不需要的组合，Skiena团队优化了错误的目标函数。

**教训**：
- 建模错误比算法错误代价更高
- 隐式约束来自对真实业务的理解
- 验证建模比优化算法更重要

## LLM时代：建模是人的价值

### LLM擅长什么

LLM擅长：
- 生成图算法代码（BFS、Dijkstra、Prim等）
- 实现数据结构（邻接表、邻接矩阵）
- 转换图表示（矩阵→邻接表）

### LLM不擅长什么

LLM不擅长：
- 判断问题是否应该建模为图
- 识别隐式约束
- 选择正确的图类型和算法

### 人的价值

**建模能力是人的核心竞争力**：

| 任务 | LLM | 人 |
|------|-----|----|
| 理解问题 | 不擅长（无法理解业务） | 擅长（理解隐式约束） |
| 识别图结构 | 不擅长（无法判断） | 擅长（识别节点和边） |
| 选择算法 | 辅助（可枚举选项） | 擅长（权衡判断） |
| 实现算法 | 擅长（生成代码） | 辅助（验证实现） |
| 验证建模 | 不擅长（无法判断） | 擅长（检查建模正确性） |

**人机协同的正确分工**：
- 人负责：理解问题、识别约束、构建模型、验证建模
- LLM负责：生成代码、枚举方案、验证实现

如果让人去做枚举而让LLM去理解问题，就颠倒了两者的擅长领域。

## 变式练习

### 练习1：建模会议排期

问题：10个会议室，每天8小时，每小时一个时间段。30场会议，每场会议需要特定会议室和时间段。目标是最大化排期会议数。

建模：
1. 节点：会议、时间段、会议室
2. 边：可用性
3. 图类型：流网络
4. 算法：最大流

验证建模：是否考虑了时间约束？

### 练习2：建模航班路线优化

问题：航空公司需要优化航班路线，最小化总飞行距离，同时覆盖所有目的地。

建模：
1. 节点：机场
2. 边：航班路线
3. 权重：飞行距离
4. 图类型：无向加权图
5. 算法：MST

验证建模：是否考虑了航班方向？（有向图还是无向图？）

### 练习3：建模社交网络推荐

问题：推荐"可能认识的人"，标准是共同好友数量。

建模：
1. 节点：用户
2. 边：好友关系
3. 权重：共同好友数
4. 图类型：无向图
5. 算法：BFS（找第2层朋友）+ 共同好友计算

验证建模：是否考虑了好友关系的双向性？

## 本节小结

### 核心问题回答

**问题**：如何将现实问题建模为图问题？建模错误的代价是什么？

**答案**：
- **建模五问法**：节点是什么？边是什么？权重是什么？图类型？标准算法？
- **建模错误代价更高**：解错了问题，所有优化都是浪费
- **LLM不能替代建模**：建模是人的价值

### 关键洞察

1. **建模比算法更重要**：算法错误可修正，建模错误是方向性错误
2. **隐式约束容易遗漏**：现实问题有隐式约束，建模时容易忽略
3. **验证建模比优化算法更重要**：先验证建模正确，再优化算法

### LLM时代视角

**核心警示**：LLM能做算法实现，但建模是人的工作。

- 让LLM尝试建模一个复杂问题，评价其建模是否正确
- 建模错误比算法错误更致命
- 人机协同的正确分工：人建模，LLM实现

### 思考题

1. 建模五问法是什么？如何应用？

### 问题情境与直觉

1. **直觉建立**：给定一个会议排期问题，尝试建模为图问题。

2. **节点识别**：会议排期问题的节点是什么？边是什么？

4. **建模练习**：将"航班路线优化"建模为图问题，确定节点、边、权重、算法。

5. **建模验证**：给定一个建模方案，验证是否正确。如果不正确，指出遗漏的约束。

6. **建模转换**：将建模转换为邻接表表示，实现图结构。

### 错误诊断与反例

7. **反例构造**：构造一个建模错误的案例（如小明最初把会议室建模为节点），展示后果。

8. **错误诊断**：某人建模会议排期，忽略时间约束。会导致什么问题？如何修正？

9. **边界条件**：如果问题不适合建模为图问题，如何判断？

### 方案比较与权衡

10. **对比分析**：对比不同建模方案（如会议排期可以建模为流网络或匹配问题），选择最优。

11. **选择决策**：以下问题选择哪种建模模式：
    - 课程排课
    - 地图导航
    - 社交网络推荐

### 变式与边界条件

12. **变式应用**：多仓库物流配送问题，如何建模？隐式约束是什么？

13. **建模边界**：某些问题不适合建模为图问题（如纯数值优化）。如何判断？

14. **隐式约束识别**：给定问题描述，识别隐式约束。

### 开放设计

15. **建模系统**：设计一个建模辅助系统，支持：
    - 输入问题描述
    - 输出建模方案（节点、边、算法）
    - 提示隐式约束

### 综合应用

16. **真实系统建模**：建模一个真实的系统（如课程排课系统、物流配送系统），验证建模正确性。

### LLM协同实验

17. **让LLM建模**：让LLM尝试建模一个复杂问题（会议排期）。评价其建模是否正确，是否遗漏隐式约束。

18. **让LLM解释**：让LLM解释为什么某个建模是错误的。观察其是否理解建模错误的代价。

19. **人机协同**：人负责建模，LLM负责实现。实践这种协同模式，评价效率。
