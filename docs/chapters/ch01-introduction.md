# 第1章 当你让 AI 帮你写代码时，你还需要懂算法吗？

---

## 导入：一个真实的故事

2024年，某互联网公司的工程师小李接到了一个任务：优化推荐系统的响应时间。系统在高峰期响应变慢，用户体验受影响。

小李打开 Cursor，输入提示词：

> "帮我优化这个推荐函数，现在太慢了"

Cursor 很快给出了一个"优化版"，用了更现代的 Python 语法，加了一些类型注解。小李运行了一下，测试用例通过了，就提交了代码。

一周后，高峰期来了，系统比以前更慢了。

问题出在哪里？Cursor 给的代码只是"看起来更现代"，但算法复杂度从 O(n log n) 变成了 O(n²)。小李没有发现，因为他不知道如何判断一个算法的复杂度——他只知道"代码能跑"。

这个故事不是个案。当 coding agent 成为标配，**"能写代码"不再是核心竞争力，"能判断代码对不对"才是。**

## 1.1 你面对的是一个什么问题

### 1.1.1 问题的三个层次

当你拿到一个任务，不管是自己写还是让 agent 写，你首先要回答三个问题。

**第一层：输入是什么？输出是什么？**

这听起来简单，但很多人连这一步都说不清楚。

> [插图位置：一个模糊的问题描述框，箭头指向多个不同的"解读"，展示同一个描述可能导致完全不同的实现]

举个例子。产品经理说：

> "帮我写一个排序程序"

这个描述够清楚吗？你至少需要问：

| 问题 | 影响 |
|------|------|
| 排序什么？整数？字符串？对象？ | 决定比较方式 |
| 多大规模？10个？100万个？ | 决定是否需要O(n)算法 |
| 数据有什么特点？基本有序？大量重复？ | 决定是否可以用特殊优化 |
| 需要稳定吗？相同元素保持原顺序？ | 排除某些算法 |
| 需要原地吗？内存有限制吗？ | 排除某些算法 |
| 实时性要求？批处理？ | 决定能否接受高复杂度 |

这些问题不问清楚，agent 给你的代码可能完全不是你要的。

> [交互可视化位置：一个"问题描述澄清器"。左边输入模糊描述，右边展示需要澄清的问题列表，点击每个问题可以看到不同的回答如何影响算法选择]

**第二层：问题的结构是什么？**

问题可以按结构分类：

| 结构类型 | 特征 | 典型问题 |
|----------|------|----------|
| 搜索问题 | 在一堆东西里找目标 | 查找、匹配、检索 |
| 优化问题 | 找"最好"的解 | 最短路径、最大收益、最小代价 |
| 判定问题 | 回答是/否 | 图是否连通？是否存在满足条件的解？ |
| 构造问题 | 生成满足条件的对象 | 排列、组合、路径 |

结构决定了方法。贪心对某些优化问题有效，对另一些则完全不行。动态规划适合有重叠子问题的情况，但不是所有问题都有这个结构。

**第三层：问题的约束是什么？**

| 约束类型 | 问题示例 |
|----------|----------|
| 时间约束 | "100万条数据，1秒内出结果" |
| 空间约束 | "内存只有512MB" |
| 精度约束 | "误差不超过0.1%" |
| 实时性约束 | "用户输入后立即响应" |

这些约束决定了 O(n²) 是可接受还是不可接受。

> [插图位置：约束与复杂度的关系图。横轴是数据规模，纵轴是操作次数，不同约束画成水平线，展示何时会突破约束]

### 1.1.2 一个完整的例子：任务调度

让我们用一个具体问题来演示如何分析。

**场景**：你有 n 个任务，每个任务有一个执行时间。你需要在单台机器上安排它们的执行顺序，使得总完成时间（所有任务的等待时间之和）最小。

**第一步：理解输入输出**

- 输入：n 个任务，每个任务有时间 t_i
- 输出：一个执行顺序（排列）
- 目标：最小化 Σ(完成时间)

**第二步：分析问题结构**

这是一个**优化问题**——要找"最好"的顺序。

这是一个**排列问题**——解是一个顺序。

这是一个**贪心可能有效的问题**——直觉上，短任务先做可能更好？

**第三步：检查约束**

如果 n=10，所有排列都试一遍（O(n!)）也没问题。
如果 n=1000000，O(n!) 绝对不行，需要 O(n log n) 或 O(n)。

**第四步：验证贪心假设**

短任务优先是否正确？让我们证明或反驳。

假设有两个任务 A(时间3) 和 B(时间5)。
- 顺序 AB：A完成时间3，B完成时间3+5=8，总和11
- 顺序 BA：B完成时间5，A完成时间5+3=8，总和13

AB 更好。短任务优先似乎对。

但如果有依赖关系呢？"任务A必须在任务B之前完成"？那就不能随便排了。

> [交互可视化位置：任务调度模拟器。拖拽任务调整顺序，实时显示总完成时间。可以添加依赖关系约束，观察贪心策略是否仍然有效]

## 1.2 如何系统化地分析问题

### 1.2.1 六问诊断法

Skiena 提出的六问法是一个系统化的问题分析框架。这不是教你"怎么写代码"，而是教你**如何思考一个问题**。

> [插图位置：六问法的循环图，每个问题是一个节点，箭头连接形成循环]

**问题 1：我真的理解问题吗？**

检验方法：
- 能否用自己的话复述？
- 能否举出 3 个不同的例子？
- 边界情况是什么？（空输入、单元素、极端值）
- 能否反例？（什么情况下问题无解？）

**问题 2：简单算法够用吗？**

暴力解是什么？
- 枚举所有可能 → 选择最优
- 复杂度多少？
- 对于你的数据规模，够不够？

> [截图位置：在 Cursor 中输入"用暴力方法解决这个问题"，展示 agent 给出的代码，以及复杂度分析]

很多时候，暴力解就够了。不要过早优化。

**问题 3：这个问题有标准解法吗？**

算法问题可以归类：

| 问题类别 | 典型算法 | 识别特征 |
|----------|----------|----------|
| 排序 | 快排、归并、堆排 | 需要有序 |
| 查找 | 二分、哈希、BST | 需要定位 |
| 图遍历 | BFS、DFS | 有关系/连接 |
| 最短路 | Dijkstra、Bellman-Ford | 路径代价 |
| 最小生成树 | Kruskal、Prim | 连通+最小代价 |
| 动态规划 | 各种 DP | 重叠子问题 |
| 贪心 | 各种贪心 | 局部最优→全局最优？ |
| 流 | Ford-Fulkerson | 容量、流量 |

**问题 4：特殊情况能否简化？**

- 某个参数固定，问题是否变简单？
- 数据有特殊结构（有序、无重复），能否利用？
- 约束放宽（允许近似解），问题是否变简单？

**问题 5：哪种算法范式适合？**

这需要你熟悉各种范式的特征：

| 范式 | 适用场景 | 典型特征 |
|------|----------|----------|
| 分治 | 可以分解为独立子问题 | 递归结构 |
| 贪心 | 局部最优导致全局最优 | 单调性、无后效性 |
| 动态规划 | 有重叠子问题 | 最优子结构 |
| 回溯 | 需要枚举但可剪枝 | 解空间树 |
| 网络流 | 匹配、分配问题 | 容量约束 |

**问题 6：还是想不出来？**

回到第一步。或者：
- 搜索类似问题
- 问更有经验的人
- 放宽约束，先解决简化版

### 1.2.2 六问法的本质

六问法本身就是一个**算法**：

```
输入：一个模糊的问题描述
输出：一个清晰的解决方案路径

while 未解决:
    1. 检查是否真正理解 → 不理解则澄清
    2. 尝试简单解 → 够用则停止
    3. 查找标准解 → 找到则应用
    4. 尝试简化 → 简化后重新分析
    5. 选择范式 → 应用范式
    6. 循环重试
```

这正是 agent 做不到的——agent 可以帮你实现，但不能帮你"理解"。

> [交互可视化位置：六问法决策树。从根节点"问题描述"开始，每个问题是一个分支，最终指向推荐的算法范式或解决方案]

## 1.3 如何验证一个解是正确的

### 1.3.1 两件武器：循环不变式与反例

当 agent 给你一段代码，你怎么判断它对不对？

**武器一：循环不变式**

循环不变式是一个在循环的每次迭代前后都保持为真的命题。用它证明算法正确的三步：

1. **初始化**：循环开始前，不变式成立
2. **保持**：假设某次迭代前不变式成立，证明迭代后仍成立
3. **终止**：循环结束时，不变式给出期望的结果

> [交互可视化位置：插入排序的循环不变式演示。显示数组状态、当前处理元素、已排序区间，逐步展示三步证明]

举个例子，插入排序的循环不变式：

> 在每次外层循环迭代开始时，A[1..j-1] 包含原来在 A[1..j-1] 中的元素，且已排序。

**证明**：
- 初始化：j=2 时，A[1..1] 只有一个元素，自然有序
- 保持：假设 A[1..j-1] 有序，内层循环将 A[j] 插入正确位置，A[1..j] 有序
- 终止：j=n+1 时，A[1..n] 有序

**武器二：反例**

反例是证明算法错误的最快方法。找一个让算法失败的输入。

> [插图位置：反例构造的思维导图。中心是"算法输出错误"，分支是"什么输入会导致错误"]

常见的反例类型：
- 边界情况：空输入、单元素、最小/最大值
- 特殊结构：已排序、逆序、全相同
- 对称性破坏：贪心策略常在需要"回溯"的地方失败

### 1.3.2 实践：检验一个 agent 生成的排序实现

假设你让 agent 实现一个排序算法，它给了你这段代码：

```python
def sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(i + 1, n):
            if arr[i] > arr[j]:
                arr[i], arr[j] = arr[j], arr[i]
    return arr
```

**第一步：识别算法**

这是选择排序。每次外层循环选出最小元素放到当前位置。

**第二步：检查边界情况**

| 测试用例 | 结果 | 正确？ |
|----------|------|--------|
| [] | [] | ✓ |
| [1] | [1] | ✓ |
| [3, 1, 2] | [1, 2, 3] | ✓ |
| [2, 2, 1] | [1, 2, 2] | ✓ |
| [3, 2, 1] | [1, 2, 3] | ✓ |

基本测试通过。

**第三步：分析复杂度**

外层循环 n 次，内层循环平均 n/2 次，总比较次数 n(n-1)/2。

复杂度 O(n²)。

**第四步：判断是否满足需求**

如果 n ≤ 10000，O(n²) 可能还行。
如果 n = 1000000，需要 5×10¹¹ 次比较，肯定太慢。

> [截图位置：在 Python 中运行这段代码，测量不同规模的运行时间，绘制图表]

**第五步：考虑是否需要更快的算法**

如果 n 很大，你需要告诉 agent：

> "这个复杂度太高，给我一个 O(n log n) 的实现"

然后检验它给你的是不是真的 O(n log n)。

> [交互可视化位置：复杂度比较器。输入数据规模，显示 O(n²) 和 O(n log n) 的预期运行时间，帮助判断哪种算法合适]

## 1.4 复杂度：你的约束，算法能否满足

### 1.4.1 增长量级：为什么 O(n log n) 和 O(n²) 差这么多

常见复杂度的增长：

| 复杂度 | n=10 | n=100 | n=1000 | n=10000 | n=100000 |
|--------|------|-------|--------|---------|----------|
| O(1) | 1 | 1 | 1 | 1 | 1 |
| O(log n) | 3 | 7 | 10 | 13 | 17 |
| O(n) | 10 | 100 | 1000 | 10000 | 100000 |
| O(n log n) | 33 | 664 | 9966 | 132877 | 1660964 |
| O(n²) | 100 | 10000 | 10⁶ | 10⁸ | 10¹⁰ |
| O(2ⁿ) | 1024 | 10³⁰ | 10³⁰¹ | - | - |

> [交互可视化位置：复杂度增长曲线图。可以调整 n 的范围，观察不同复杂度的增长速度]

**关键洞察**：从 O(n²) 到 O(n log n)，不是"快一点"，而是**从不可能到可能**。n=1000000 时，O(n²) 是 10¹² 次操作（假设每次操作 1 纳秒，需要 17 分钟），O(n log n) 是 2×10⁷ 次（20 毫秒）。

### 1.4.2 一个常见的误解

"这个算法是 O(n log n) 的，所以它比 O(n²) 快。"

**不对。**

n 小的时候，O(n²) 可能更快——因为常数因子。

快排平均 O(n log n)，但常数因子小，对小数组反而比归并排序快。很多快排实现在 n < 10 时会切换到插入排序。

> [插图位置：不同算法的实际运行时间比较图。展示小规模数据时 O(n²) 的插入排序可能比 O(n log n) 的快排更快]

渐进分析告诉你的是**当 n 趋向无穷大时**的趋势，不是对所有 n 都成立。

### 1.4.3 你的约束是什么

当 agent 给你一个 O(n²) 的实现，你需要判断：

| 数据规模 | 时间限制 | O(n²) 可接受？ |
|----------|----------|----------------|
| n ≤ 1000 | 秒级 | 可能可以 |
| n ≤ 10000 | 秒级 | 勉强可以 |
| n = 100000 | 秒级 | 不行 |
| n = 1000000 | 秒级 | 绝对不行 |

> [交互可视化位置：约束-复杂度决策器。输入数据规模和时间限制，输出可接受的复杂度上限]

## 1.5 建模：把现实问题翻译成算法语言

### 1.5.1 什么是建模

建模是把一个现实问题翻译成算法语言的过程。

**现实问题** → **算法问题** → **已知解法**

| 现实问题 | 算法问题 | 解法 |
|----------|----------|------|
| 课程排课（先修关系） | 有向图的拓扑排序 | Kahn / DFS |
| 最短路线 | 图的最短路径 | Dijkstra / BFS |
| 任务分配 | 二分图匹配 / 最大流 | 匈牙利 / Ford-Fulkerson |
| 背包打包 | 背包问题 | DP |
| 数据压缩 | 哈夫曼编码 | 贪心 |

建模失误比算法失误代价更高。如果你把问题建模错了，最优的算法解的也是错的问题。

### 1.5.2 常见的建模对象

**排列（Permutation）**

适用：顺序问题
例子：旅行商（访问顺序）、调度（执行顺序）

```
排列问题特征：解是一个顺序，排列数量 n!
```

**子集（Subset）**

适用：选择问题
例子：背包（选哪些物品）、子集和（选哪些数）

```
子集问题特征：解是一个选择，子集数量 2ⁿ
```

**图（Graph）**

适用：关系问题
例子：网络（节点和连接）、依赖（有向边）

```
图问题特征：有关系/连接，顶点和边
```

**树（Tree）**

适用：层次问题
例子：组织结构、语法树、决策树

```
树问题特征：有层次/包含关系，无环
```

> [交互可视化位置：建模对象选择器。展示一个现实问题，让用户选择合适的建模对象，然后显示对应的算法类别]

### 1.5.3 建模是人的核心能力

为什么 agent 帮不了你建模？

因为建模需要你理解问题的**本质**，而不仅仅是表面描述。

"课程排课"这个描述：
- 如果问的是"哪些课可以同时上" → 图的着色问题
- 如果问的是"按什么顺序上才能满足先修关系" → 拓扑排序
- 如果问的是"如何安排教室和时间避免冲突" → 约束满足问题

同一个描述，可能对应完全不同的算法问题。**这个判断，只有人能做。**

**练习 1.3**：复杂度判断

> 给定一个算法的代码，判断它的最坏时间复杂度

```python
def mystery(arr):
    n = len(arr)
    result = []
    for i in range(n):
        inserted = False
        for j in range(len(result)):
            if arr[i] < result[j]:
                result.insert(j, arr[i])
                inserted = True
                break
        if not inserted:
            result.append(arr[i])
    return result
```

**练习 1.7**：建模练习

2. 识别问题结构（搜索/优化/判定/构造）
3. 归类到已知算法类别
4. 你的描述足够清晰到让 agent 理解吗？

---

## 1.7 小结

这一章没有教你怎么写代码。它教的是：

| 能力 | 描述 |
|------|------|
| 描述 | 把模糊的需求变成清晰的问题陈述 |
| 分析 | 识别问题的结构和约束 |
| 验证 | 判断一个解是否正确 |
| 判断 | 评估复杂度是否满足需求 |
| 建模 | 把现实问题翻译成算法语言 |

这些是 coding agent 做不到的事。

在接下来的章节中，我们会看到具体的算法——排序、查找、图、动态规划……但你学习的重点不是"怎么实现"，而是：

- 这个算法解决什么问题？
- 它的正确性如何证明？
- 它的复杂度是多少？
- 什么情况下应该用它？
- 如何判断 agent 给出的实现是否正确？

**算法思维，比算法实现更重要。**
