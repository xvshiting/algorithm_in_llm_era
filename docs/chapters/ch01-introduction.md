# 第1章 问题建模与算法思维

## 写在前面

这一章要建立的是一种思维方式。在 coding agent 能帮你实现代码的时代，你还需要学算法吗？答案是：**你需要学的不是"怎么写"，而是"怎么判断"**。

判断什么？判断问题的结构是什么，判断该用什么方法，判断 agent 给出的实现是否正确，判断复杂度是否合理，判断边界情况有没有被遗漏。这些判断能力，agent 帮不了你——因为它不知道你在解决什么问题，它只知道你给了它什么描述。

这一章的核心：**建模是人的工作，实现是 agent 的工作。但你得先能把问题说清楚。**

---

## 1.1 你面对的是一个什么问题

算法课程通常从"什么是算法"开始。但更重要的问题是：**你面对的是一个什么问题？**

### 问题的三个层次

**第一层：输入是什么？输出是什么？**

这听起来简单，但很多人连这一步都说不清楚。"帮我写一个排序程序"——排序什么？多大规模？数据有什么特点？稳定吗？原地吗？这些约束不明确，agent 给你的代码可能完全不是你要的。

**第二层：问题的结构是什么？**

这是一个搜索问题？优化问题？判定问题？是离散的还是连续的？是精确解还是近似解？结构决定了方法——贪心可能对某些问题有效，对另一些则完全不行。

**第三层：问题的本质约束是什么？**

时间限制？空间限制？数据规模？实时性要求？这些约束决定了你能否接受一个 O(n²) 的解，还是必须追求 O(n log n) 甚至 O(n)。

### 一个例子：任务调度

"帮我安排一下这些任务的执行顺序，让总完成时间最短。"

这个描述够清楚吗？不够。你需要问：

- 任务之间有依赖关系吗？（有依赖 → 拓扑排序；无依赖 → 可能是调度问题）
- 有多台机器吗？（单机 → 简单；多机 → NP 困难）
- 任务有到达时间吗？（有 → 更复杂；无 → 相对简单）
- 你要的是最优解还是足够好的解？

这些问题不问清楚，agent 给你的代码可能解决的是完全不同的问题。

## 1.2 如何描述一个问题

### 六问诊断法

Skiena 提出的六问法是一个系统化的问题分析框架：

1. **我真的理解问题吗？** — 能否用自己的话复述？能否举几个例子？边界情况是什么？
2. **简单算法够用吗？** — 暴力解是什么？复杂度多少？对小规模数据够不够？
3. **这个问题有标准解法吗？** — 能否归入已知的算法类别？
4. **特殊情况能否简化？** — 如果某些参数固定，问题是否变简单？
5. **哪种算法范式适合？** — 分治？贪心？动态规划？图算法？搜索？
6. **还是想不出来？** — 回到第一步，或者换一个角度

### 六问法的本质

六问法不是教你"怎么写代码"，而是教你**如何思考一个问题**。这正是 agent 做不到的——agent 可以帮你实现，但不能帮你"理解"。

这个方法本身就是一个算法：输入一个模糊的问题描述，输出一个清晰的解决方案。在最后一章，我们会看到如何把这个方法编码成一个 agent skill。

## 1.3 如何验证一个算法是正确的

### 两件武器：循环不变式与反例

**循环不变式**证明算法正确。三步：初始化（循环开始前成立）、保持（每次迭代保持成立）、终止（循环结束时得到期望结果）。

**反例**证明算法错误。找一个让算法失败的输入，比写一个正确的证明更容易——但需要你对算法的逻辑有清晰理解。

### 为什么这两个能力在 agent 时代更重要？

因为 agent 会给你"看起来对"的代码。它甚至可能通过大部分测试用例。但边界情况呢？空输入？单元素？已排序？逆序？重复元素？这些 agent 可能想不到。

你需要的是**判断代码正确性的能力**，不是写代码的能力。循环不变式和反例构造，就是你的判断工具。

### 实践：检验一个排序实现

假设你让 agent 实现了一个排序算法。你怎么判断它对不对？

```
def sort(arr):
    # agent 给出的实现
    ...
```

你应该问的问题：

1. 空数组会怎样？单元素数组呢？
2. 已排序的数组会怎样？逆序的呢？
3. 有重复元素会怎样？
4. 循环不变式是什么？能写下来吗？
5. 最坏情况复杂度是多少？有没有可能退化？

这些问题 agent 不会主动帮你问——你必须自己问。

## 1.4 复杂度：你需要的性能，算法能给你吗

### 增长量级

O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ)

这不是微小的差异——这是质变。n=10⁶ 时，O(n²) 是 10¹² 次操作，O(n log n) 是 2×10⁷ 次，差 5 个数量级。

### 你的约束是什么？

- 数据规模多大？10³？10⁶？10⁹？
- 时间限制多严？毫秒级？秒级？分钟级？
- 空间限制多大？内存放得下吗？需要外存吗？

这些约束决定了 O(n²) 是可接受还是不可接受。agent 给你的代码可能是 O(n²) 的——它不知道你的数据规模。你得自己判断。

### 一个常见的错误

"这个算法是 O(n log n) 的，所以它比 O(n²) 快。"

不对。n 小的时候，O(n²) 可能更快——因为常数因子。渐进分析告诉你的是**当 n 趋向无穷大时**的趋势，不是对所有 n 都成立。

## 1.5 建模：把现实问题变成算法问题

### 什么是建模

建模是把一个现实问题翻译成算法语言的过程。"我有一堆任务要安排" → "这是一个调度问题"。"我要找两个城市间的最短路" → "这是一个图上的最短路径问题"。

建模失误比算法失误代价更高。如果你把问题建模错了，最优的算法解的也是错的问题。

### 常见的建模对象

- **排列**：顺序问题（旅行商、调度）
- **子集**：选择问题（背包、子集和）
- **图**：关系问题（网络、依赖）
- **树**：层次问题（组织结构、语法）
- **网格**：空间问题（路径规划、图像）

### 建模是人的核心能力

为什么 agent 帮不了你建模？因为建模需要你理解问题的**本质**，而不仅仅是表面描述。"课程排课"可以是图问题（先修关系），可以是调度问题（时间冲突），可以是约束满足问题——取决于你问的是什么。

这个判断，只有人能做。

## 1.6 这一章的练习

### 基础练习

1. 用循环不变式证明：在一个已排序数组中插入一个元素后，数组仍然有序。
2. 构造反例：证明"每次选价值最高的物品"这个贪心策略在 0-1 背包问题上不正确。
3. 分析复杂度：给定一个算法的代码，判断它的最坏时间复杂度。

### 进阶练习

4. **问题描述练习**：找一个你熟悉的问题（如"安排会议时间"），用六问法完整分析一遍。然后让 agent 帮你实现，看看你的描述是否足够清晰。

5. **正确性检验练习**：让 agent 实现一个二分查找。然后你用循环不变式检验它的正确性，并构造测试用例验证边界情况。

6. **复杂度判断练习**：给定一个问题的规模约束（如"最多 10⁶ 个元素，需要在 1 秒内完成"），判断 O(n log n) 是否足够，还是需要 O(n)。

### 挑战练习

7. **建模练习**：描述一个你日常工作中的问题，将其建模为某种算法问题。你的描述足够清晰吗？agent 能理解吗？如果不能，问题出在哪里？

## 1.7 小结

这一章没有教你怎么写代码。它教的是：

- 如何**描述**一个问题
- 如何**分析**问题的结构
- 如何**验证**一个解是否正确
- 如何**判断**复杂度是否可接受
- 如何**建模**把现实翻译成算法

这些是 coding agent 做不到的事。在接下来的章节中，我们会看到具体的算法——但你学习的重点不是"怎么实现"，而是"怎么理解、怎么选择、怎么验证"。
