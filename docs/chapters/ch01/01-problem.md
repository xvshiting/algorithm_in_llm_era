# 1.1 你面对的是一个什么问题

当你拿到一个任务，不管是自己写代码还是让 agent 帮你写，你首先要回答的不是"用什么算法"，而是"我在解决什么问题"。

这听起来很简单。但很多人连这一步都做不好。他们给 agent 一个模糊的描述，得到一个"能跑"的代码，却不知道这个代码解决的是不是自己想要的问题。

---

## 问题的第一层：输入和输出

最基本的，你需要说清楚输入是什么、输出是什么。

产品经理说："帮我写一个排序程序。"

这个描述够清楚吗？不够。你需要问清楚很多事情。

排序什么？是整数、字符串、还是复杂的对象？如果是对象，按什么字段排序？如果字段值相同，怎么处理？

数据规模多大？是十个八个，还是百万千万？规模决定了 O(n²) 的算法能不能用。

数据有什么特点？是基本有序、完全随机、还是有很多重复？特点决定了能不能用特殊优化。

需要稳定吗？如果两个元素相等，它们在输出中的相对顺序要保持吗？有些排序算法是稳定的，有些不是。

需要原地吗？内存有限制吗？原地排序不需要额外空间，但有些算法需要。

这些问题不问清楚，agent 给你的代码可能完全不是你要的。它可能给你一个稳定的 O(n log n) 排序，但你需要的是原地排序。它可能给你一个 O(n²) 的排序，但你的数据有百万条。

---

## 一个具体例子

让我们用一个具体问题来演示如何分析。

场景是这样的：你有 n 个任务，每个任务有一个执行时间。你需要在单台机器上安排它们的执行顺序，使得"总完成时间"最小。这里的"总完成时间"指的是所有任务的完成时间之和。

举个例子。假设有三个任务，时间分别是 3、5、2。

如果按 3→5→2 的顺序执行：第一个任务 3 完成，第二个任务 3+5=8 完成，第三个任务 3+5+2=10 完成。总完成时间 = 3 + 8 + 10 = 21。

如果按 2→3→5 的顺序执行：第一个任务 2 完成，第二个任务 2+3=5 完成，第三个任务 2+3+5=10 完成。总完成时间 = 2 + 5 + 10 = 17。

第二个顺序更好。为什么？因为短任务先做，后面等待的任务就少了。

这个问题叫做"最小化总完成时间的单机调度问题"。它有一个很简单的最优解法：按执行时间从小到大排序。这就是所谓的"短作业优先"策略。

但事情没这么简单。如果任务之间有依赖关系呢？比如"任务 A 必须在任务 B 之前完成"？那就不能随便排序了。这时问题变成了一个拓扑排序问题，而不是简单的排序问题。

同一个"任务调度"的描述，可能对应完全不同的算法问题。

---

## 问题的第二层：结构是什么

问题可以按结构分类。识别结构是选择算法的第一步。

搜索问题是在一堆东西里找目标。比如查找一个数、匹配一个字符串、检索符合条件的记录。搜索问题的解是一个位置或者"不存在"。

优化问题是找"最好"的解。比如最短路径、最大收益、最小代价。优化问题的解是一个最优的方案，可能有多个等价的最优解。

判定问题是回答"是"或"否"。比如图是否连通、是否存在满足条件的解。判定问题的解是一个布尔值。

构造问题是生成满足条件的对象。比如输出一个排列、构造一个路径、生成一个满足约束的方案。构造问题的解是一个具体的对象。

结构决定了方法。贪心对某些优化问题有效，对另一些则完全不行。动态规划适合有重叠子问题的情况，但不是所有问题都有这个结构。

---

## 问题的第三层：约束是什么

约束决定了什么是"可接受"的解。

时间约束：必须在多少时间内出结果？1秒？1分钟？1小时？

空间约束：内存有限制吗？需要存到磁盘吗？

精度约束：需要精确解还是近似解？误差可以接受多少？

实时性约束：是批处理还是实时响应？用户在等待吗？

这些约束决定了 O(n²) 是可接受还是不可接受。如果数据规模是一百万，O(n²) 是一万亿次操作。假设每次操作 1 纳秒，也需要 17 分钟。如果要求 1 秒内出结果，O(n²) 绝对不行。

但如果数据规模是一千，O(n²) 是一百万次操作，1 毫秒就搞定了，完全没问题。

## 当你描述问题给 agent 时

回到我们的核心场景：你让 coding agent 帮你写代码。

最常见的问题是描述不够清晰。你说"帮我优化这个函数"，agent 不知道你的约束是什么。你说"写一个高效的排序"，agent 不知道你的数据规模是什么。

好的问题描述应该包含这些要素：

输入是什么，格式是什么，范围是什么。

输出是什么，格式是什么，有什么要求。

约束是什么，时间限制、空间限制、精度要求。

特殊情况是什么，空输入怎么办、边界值怎么办。

你可以把这些问题整理成一个模板，每次让 agent 写代码之前先填一遍。这不是浪费时间，而是避免返工。
