# 1.2 六问诊断法

当你面对一个算法问题时，从哪里开始？

很多人的反应是直接开始写代码。但在 agent 时代，直接让 agent 写代码往往得到一个"能跑但不是你想要的"结果。你需要先想清楚问题，再让 agent 实现。

Skiena 提出的六问诊断法是一个系统化的思考框架。它不是教你写代码，而是教你如何分析问题。这个方法本身就是一个算法：输入一个模糊的问题，输出一个清晰的解决方案路径。

---

## 第一问：我真的理解问题吗？

检验是否真正理解一个问题的最好方法是：你能不能用自己的话复述一遍？

如果产品经理说"帮我写一个推荐算法"，你能复述成"给定用户的历史行为，预测用户可能感兴趣的物品并排序输出"吗？这个复述已经比原始描述清晰多了。

但还不够。你还需要举出具体的例子。输入是什么样子的？输出是什么样子的？正常情况是什么？边界情况是什么？

边界情况经常被忽略，却是最容易出问题的地方。空输入怎么办？单元素怎么办？极端值怎么办？重复元素怎么办？这些情况 agent 可能想不到，你得帮它想到。

---

## 第二问：简单算法够用吗？

很多人一上来就想"最优解"。但对于很多问题，暴力解就够了。

暴力解是什么？枚举所有可能，选择最好的一个。复杂度通常很高，可能是 O(n²)、O(n!)、甚至指数级。但如果你的数据规模很小，高复杂度也没关系。

举个例子。你需要从 10 个人中选出 3 个人组成一个团队。枚举所有组合是 C(10,3) = 120 种，眨眼就跑完了。不需要什么高级算法。

但如果是从 1000 个人中选 500 个呢？C(1000,500) 是一个天文数字，暴力解绝对不行。

判断简单算法是否够用，需要你清楚自己的数据规模。如果 agent 给你一个 O(n²) 的解，你需要判断：我的 n 是多少？O(n²) 能不能接受？

---

## 第三问：这个问题有标准解法吗？

很多问题其实是有标准解法的。你不需要重新发明轮子，只需要识别问题类型，然后套用标准算法。

排序问题：快排、归并、堆排。查找问题：二分、哈希。图遍历：BFS、DFS。最短路：Dijkstra、Bellman-Ford。最小生成树：Kruskal、Prim。动态规划、贪心、网络流……这些算法都是几十年甚至几百年智慧的结晶。

识别问题类型需要你熟悉各种算法的"形状"。排序问题的形状是"让一堆东西有序"。查找问题的形状是"在一堆东西里定位目标"。图问题的形状是"有关系、有连接"。

当你识别出问题类型，就可以告诉 agent："这是一个最短路径问题，请用 Dijkstra 算法实现。"而不是给它一个模糊的描述。

---

## 第四问：特殊情况能否简化？

有时候，问题的某些特殊版本更容易解决。

如果某个参数固定，问题是否变简单？比如背包问题，如果容量固定，可以用更简单的 DP。

如果数据有特殊结构，能否利用？比如排序问题，如果数据范围有限，可以用计数排序，O(n) 搞定。

如果约束放宽，问题是否变简单？比如"最优解"变成"足够好的解"，可能从 NP 困难变成多项式时间可解。

这些简化思路可以帮助你找到突破口，或者至少找到一个"可以接受"的解。

---

## 第五问：哪种算法范式适合？

算法有几种基本范式：分治、贪心、动态规划、回溯、网络流。每种范式适合一类问题。

分治适合可以分解为独立子问题的情况。归并排序把数组分成两半，分别排序，再合并。子问题是独立的。

贪心适合"局部最优导致全局最优"的情况。最小生成树的 Kruskal 算法，每次选最短的边，最后得到的就是最优解。但贪心不是万能的，很多问题局部最优不等于全局最优。

动态规划适合有重叠子问题的情况。斐波那契数列，F(n) = F(n-1) + F(n-2)，子问题会重复计算。用表格存储中间结果，可以大幅提高效率。

回溯适合需要枚举但可以剪枝的情况。八皇后问题，逐行放皇后，如果发现冲突就回溯，不需要枚举所有可能性。

网络流适合匹配、分配问题。二分图匹配、最大流、最小割都可以用网络流框架解决。

选择范式需要你对问题的结构有清晰认识。这不是 agent 能帮你做的判断。

---

## 第六问：还是想不出来？

如果五问之后还是卡住了，不要硬想。

回到第一步，重新审视问题。也许你对问题的理解有偏差。

搜索类似问题。别人可能已经解决过类似的问题。

问更有经验的人。算法社区很活跃，Stack Overflow、知乎、论文都是很好的资源。

放宽约束。先解决简化版的问题，再逐步加约束。

六问法是一个循环，不是线性流程。想不出来就回到起点，换个角度再分析。

---

## 六问法的本质

六问法不是一个检查清单，而是一个思考习惯。

养成这个习惯，你会发现：大部分时候，问题出在第一步——你根本没理解问题是什么。当你真正理解了问题，解法往往就浮出水面了。

在最后一章，我们会看到如何把这个方法编码成一个 agent skill。但无论如何，这个思考过程本身是无法被自动化的。它是人的核心能力。
