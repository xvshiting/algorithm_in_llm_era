# 1.2 六问诊断法

## 学习目标

学完这一节，你能：
- 用六问法系统化地分析算法问题
- 判断什么时候该用暴力解，什么时候需要优化
- 识别问题类型，匹配已知算法
- 把六问法作为与 agent 协作的"认知协议"

---

试飞员在面对紧急情况时，不会恐慌。他们会跑过一张可能行动的清单：检查油量、检查仪表、尝试重启、寻找降落点……这个清单不是保证成功的魔法，而是防止瘫痪的纪律。

算法设计也一样。当你面对一个陌生问题，大脑一片空白时，你需要一张清单。Skiena 把这张清单提炼为六问诊断法。

它不是一个产出算法的公式，而是防止你"冻结"的思考纪律。

---

## 第一问：我真的理解问题吗？

很多人以为自己理解了问题，其实没有。

检验方法是：**你能不能用自己的话复述一遍？**

产品经理说"帮我写一个推荐算法"。你复述成"给定用户的历史行为，预测用户可能感兴趣的物品并排序输出"。这已经比原始描述清晰了。

但还不够。你还需要回答四个子问题：

**1. 输入是什么？输出是什么？**

输入是用户行为日志？还是用户画像？还是商品特征？输出是 Top-10 推荐？还是带分数的排序列表？

**2. 数据规模多大？速度要求多快？**

是 100 个用户还是 100 万用户？响应时间是 1 秒还是 100 毫秒？规模和速度约束直接决定算法选择。

**3. 问题的领域是什么？**

这是搜索问题？排序问题？匹配问题？优化问题？领域识别是选择算法的第一步。

**4. 能不能举一个具体例子？**

输入 `[用户A, 购买记录: [手机, 耳机]]`，期望输出什么？正常情况是什么？边界情况是什么（新用户无历史、冷启动商品）？

**与 agent 的关系**：这一问决定了你给 agent 的提示词质量。模糊的提示词得到模糊的代码。清晰的提示词得到可用的代码。

**常见误区**：跳过这一问，直接问 agent "帮我写一个推荐算法"。结果 agent 给你一个基于内容的推荐，而你想要的是协同过滤。

---

## 第二问：简单算法够用吗？

很多人一上来就想"最优解"。但对于很多问题，暴力解就够了。

**暴力解是什么**：枚举所有可能，选择最好的一个。复杂度通常很高——O(n²)、O(2^n)、甚至 O(n!)。

**什么时候暴力解够用**：

- 数据规模小。从 10 个人中选 3 个组队，C(10,3) = 120 种组合，瞬间跑完。
- 速度要求不高。离线批量处理，跑一晚上也没关系。
- 原型验证。先让暴力解跑通，再考虑优化。

**什么时候暴力解不够用**：

- 数据规模大。从 1000 人中选 500 人，C(1000,500) 是天文数字。
- 实时性要求。用户在等结果，不能跑几秒钟。
- 资源受限。嵌入式设备、移动端，算力有限。

**与 agent 的关系**：agent 可能给你一个 O(n²) 的解。你需要判断：我的 n 是多少？O(n²) 能不能接受？如果 n=100，O(n²) 才 10000 次操作，完全没问题。如果 n=100万，O(n²) 是 10^12 次操作，绝对不行。

**常见误区**：一上来就追求最优解，把问题复杂化。先用暴力解跑通，再优化。

---

## 第三问：这个问题有标准解法吗？

很多问题有标准解法。你不需要重新发明轮子。

**问题类型与标准算法**：

| 问题类型 | 标准算法 | 特征 |
|---------|---------|------|
| 排序 | 快排、归并、堆排 | 让一堆东西有序 |
| 查找 | 二分、哈希 | 在一堆东西里定位目标 |
| 图遍历 | BFS、DFS | 有关系、有连接 |
| 最短路 | Dijkstra、Bellman-Ford | 找最短/最快路径 |
| 最小生成树 | Kruskal、Prim | 连接所有节点的最小代价 |
| 匹配 | 匈牙利、最大流 | 一一对应、最优分配 |

**识别问题类型的方法**：

1. 看输入。如果是图（节点+边），往图算法想。如果是序列，往排序/查找/DP 想。
2. 看输出。要排序？要最值？要计数？要路径？
3. 看约束。有时间约束往更优算法想。有空间约束往原地算法想。

**与 agent 的关系**：识别出问题类型后，你可以直接告诉 agent "这是一个最短路径问题，请用 Dijkstra 算法实现"。而不是给它一个模糊描述让它猜。

**常见误区**：不识别问题类型，直接让 agent "写一个最优的算法"。agent 不知道你要什么，可能给你一个贪心解，而你期望的是精确解。

---

## 第四问：特殊情况能否简化？

有时候，问题的特殊版本更容易解决。

**简化策略**：

**1. 固定某个参数**

背包问题如果容量固定，状态空间从二维降为一维，复杂度大幅下降。

**2. 利用数据特殊结构**

排序问题如果数据范围有限（比如 0-100 的整数），可以用计数排序，O(n) 搞定。如果数据几乎有序，插入排序可能比快排更快。

**3. 放宽约束**

"最优解"变成"足够好的解"，可能从 NP 困难变成多项式时间可解。比如 TSP 问题，精确解是 NP 困难，但最近邻启发式可以快速得到一个还不错的解。

**4. 先简化再推广**

先解决二维情况，再推广到三维。先解决无权重版本，再加权重。每一步都比直接解决原问题简单。

**与 agent 的关系**：agent 可能给你一个通用的解，但你的问题有特殊结构。你需要判断：这个通用解能不能利用我的特殊结构来优化？

**常见误区**：忽视数据特征，套用通用算法。比如对几乎有序的数据用快排，反而比插入排序慢。

---

## 第五问：哪种算法范式适合？

算法有几种基本范式。每种范式适合一类问题。

| 范式 | 适用条件 | 典型例子 |
|------|---------|---------|
| **分治** | 可分解为独立子问题 | 归并排序、快速排序 |
| **贪心** | 局部最优导致全局最优 | 最小生成树、Huffman 编码 |
| **动态规划** | 有重叠子问题 | 最短路径、背包问题 |
| **回溯** | 需要枚举但可剪枝 | 八皇后、数独 |
| **网络流** | 匹配、分配问题 | 二分图匹配、最大流 |
| **随机化** | 允许概率性结果 | 跳表、布隆过滤器 |

**如何选择范式**：

1. **看子问题是否独立**。独立 → 分治。有重叠 → DP。
2. **看局部最优是否保证全局最优**。是 → 贪心。否 → DP 或回溯。
3. **看是否需要枚举**。枚举空间大 → 回溯+剪枝。枚举空间小 → 暴力。
4. **看是否涉及匹配/分配**。是 → 网络流。

**与 agent 的关系**：这一问是人必须做的判断。agent 可以帮你实现 DP，但它无法判断"这个问题应该用 DP 而不是贪心"。这是架构层面的决策。

**常见误区**：不管什么问题都用一个范式。比如贪心能解决的问题偏要用 DP，DP 该用的时候却用了贪心。

---

## 第六问：还是想不出来？

如果五问之后还是卡住了，不要硬想。

**Skiena 的建议**：

1. **回到第一问**。也许你对问题的理解有偏差。重新审视输入、输出、约束。
2. **搜索类似问题**。别人可能已经解决过类似的问题。Google、Stack Overflow、论文。
3. **请教专家**。算法社区很活跃，不知道就问。
4. **放宽约束**。先解决简化版，再逐步加约束。

**关键原则**：正确回答"能不能这样做"不是"不能"，而是"不能，因为……"。说出为什么不行，等于检查你是否忽略了可能性。找不到让人信服的理由，往往意味着你的结论是错的。

六问法是一个循环，不是线性流程。想不出来就回到起点，换个角度再分析。

---

## 本节要点

- 六问法是防止"冻结"的思考纪律，不是产出算法的公式。
- 第一问最重要：大部分问题出在没理解问题本身。
- 简单算法够用就不要复杂化；有标准解法就不要重新发明。
- 特殊结构可以利用；范式选择是架构决策。
- 六问法是人与 agent 协作的认知协议：人负责分析和判断，agent 负责实现。

## 课堂练习

用六问法分析下面这个问题：

> 给定一个无向图，判断其中是否存在环。

请依次回答：
1. 我真的理解问题吗？（输入、输出、例子）
2. 简单算法是什么？复杂度多少？
3. 这是哪类问题？有标准解法吗？
4. 有特殊情况可以利用吗？
5. 哪种范式适合？
6. 如果还是想不出来，怎么办？
