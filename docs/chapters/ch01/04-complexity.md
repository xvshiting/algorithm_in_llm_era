# 1.4 复杂度判断

当 agent 给你一段代码，告诉你"这个算法很快"，你信吗？

你需要有自己的判断能力。复杂度分析就是这个判断工具。它告诉你：随着数据规模增长，算法的运行时间会怎样变化。

---

## 什么是复杂度

复杂度描述的是算法运行时间或空间随数据规模增长的趋势。

我们用大 O 记号来表示。O(n) 表示时间与数据规模 n 成正比。O(n²) 表示时间与 n 的平方成正比。O(log n) 表示时间与 n 的对数成正比。

关键洞察是：复杂度描述的是增长趋势，不是绝对时间。一个 O(n) 的算法可能每次操作很慢，而一个 O(n²) 的算法可能每次操作很快。当 n 很小时，O(n²) 可能反而更快。

但当 n 趋向无穷大时，增长趋势决定了谁更快。O(n log n) 最终会打败 O(n²)，不管常数因子差多少。

---

## 常见复杂度的增长

让我们看一些具体数字。

假设每次操作需要 1 纳秒。

当 n = 1000 时，O(n) 需要 1 微秒，O(n log n) 需要 10 微秒，O(n²) 需要 1 毫秒，O(2ⁿ) 需要的时间比宇宙年龄还长。

当 n = 100万 时，O(n) 需要 1 毫秒，O(n log n) 需要 20 毫秒，O(n²) 需要 17 分钟，O(n³) 需要 32 年。

当 n = 10亿 时，O(n) 需要 1 秒，O(n log n) 需要 30 秒，O(n²) 需要 32 年。

从这些数字可以看到，复杂度的差异不是"快一点"，而是"从可能到不可能"。O(n²) 在 n=100万 时已经需要十几分钟，在很多场景下不可接受。

---

## 一个常见的误解

"这个算法是 O(n log n) 的，所以它比 O(n²) 快。"

这是一个常见的误解。大 O 描述的是增长趋势，不是所有情况下的绝对快慢。

当 n 很小时，O(n²) 可能更快。原因有两个。

第一，常数因子。O(n log n) 的算法可能每次操作很复杂，常数因子大。O(n²) 的算法可能很简单，常数因子小。

第二，硬件特性。缓存、分支预测等因素对小规模数据影响很大。理论上"更快"的算法可能在实际上更慢。

这就是为什么很多快排实现在 n < 10 时会切换到插入排序。虽然插入排序是 O(n²)，但在小数据上它更快。

但记住，这个反直觉只在 n 很小时成立。当 n 变大，增长趋势会压倒一切。

---

## 你的约束是什么

当 agent 给你一个 O(n²) 的实现，你需要的不是问"有没有更快的算法"，而是问"我的场景需要多快"。

判断需要两个信息：数据规模和时间限制。

数据规模是多少？10？1000？100万？10亿？

时间限制是多少？毫秒级？秒级？分钟级？

有了这两个信息，你就可以判断复杂度是否可接受。

一个经验法则：如果时间限制是秒级，现代计算机每秒可以执行大约 10⁸ 到 10⁹ 次基本操作。如果算法复杂度是 O(n²)，n=10000 时大约是 10⁸ 次操作，刚好处在临界点。如果 n=100000，那就是 10¹⁰ 次操作，可能需要几十秒甚至几分钟，在很多场景下不可接受。

---

## 如何判断代码的复杂度

看代码判断复杂度，需要你熟悉一些基本模式。

单层循环遍历数组，通常是 O(n)。两层嵌套循环，通常是 O(n²)。二分查找那样的"每次减半"，通常是 O(log n)。分治算法，如果每次分成两半，递归深度 O(log n)，每层处理 O(n)，总共 O(n log n)。

但有很多陷阱。有些嵌套循环看起来是 O(n²)，实际上内层循环不是每次都跑满。有些单层循环看起来是 O(n)，实际上每次迭代做了 O(n) 的工作。

你需要仔细分析循环的实际执行次数，不能只看嵌套层数。

---

## 复杂度分析的局限

复杂度分析有两个重要局限。

第一，它只看最坏情况。有些算法最坏情况很糟，但平均情况很好。快排最坏 O(n²)，但平均 O(n log n)。如果你只看最坏，可能会错过一个好算法。

第二，它忽略常数因子和低阶项。O(n) 和 O(100n) 在大 O 意义下是一样的，但实际运行时间差 100 倍。对于实际工程，常数因子很重要。

尽管有这些局限，复杂度分析仍然是判断算法性能的基本工具。它给你一个快速评估的能力，让你在写代码或让 agent 写代码之前，就知道什么复杂度是"可接受"的。

---

## 小结

复杂度分析是判断算法性能的基本工具。

你需要知道常见复杂度的增长速度，以及你的数据规模和时间限制。有了这些信息，你就可以判断一个算法是否满足需求。

记住：复杂度描述的是增长趋势，不是绝对快慢。当 n 很小时，O(n²) 可能比 O(n log n) 更快。但当 n 变大，增长趋势会压倒一切。

在 agent 时代，复杂度分析更重要。因为 agent 给你的代码可能是 O(n²) 的，它不知道你的数据规模。你得自己判断。
