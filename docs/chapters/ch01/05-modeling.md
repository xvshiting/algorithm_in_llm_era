# 1.5 建模

这一章反复强调一个观点：建模是人的核心能力，agent 做不到。

什么是建模？把现实问题翻译成算法语言。

为什么 agent 做不到？因为建模需要你理解问题的本质，而不仅仅是表面描述。同一个描述可能对应完全不同的算法问题，只有人能判断。

---

## 从现实到算法

让我们看几个例子。

课程排课问题。如果问的是"哪些课可以同时上"，这是一个图的着色问题：把课程看作顶点，冲突的课程连边，目标是给每个顶点着色，相邻顶点颜色不同，颜色数最少。如果问的是"按什么顺序上才能满足先修关系"，这是一个拓扑排序问题：把课程看作顶点，先修关系看作有向边，找一个线性顺序使得所有边都从前指向后。

同一个"课程排课"的描述，对应完全不同的算法问题。

最短路线问题。如果是步行导航，要考虑道路长度、交通状况、红绿灯，这是一个加权图的最短路径问题。如果是无人机飞行，可以直接飞直线，这是一个几何问题。如果是网络数据包路由，要考虑带宽、延迟、拥塞，这是一个网络流问题。

同一个"找最短路径"的描述，也对应不同的问题。

---

## 常见的建模对象

算法问题可以归为几类常见对象。

排列问题。解是一个顺序，排列数量是 n!。例子有旅行商（访问城市的顺序）、调度（执行任务的顺序）、排列组合。

子集问题。解是一个选择，子集数量是 2ⁿ。例子有背包（选哪些物品）、子集和（选哪些数使和等于目标）、集合覆盖。

图问题。解涉及顶点和边的关系。例子有网络（节点和连接）、依赖（有向边）、社交网络（人和关系）。

树问题。解涉及层次结构。例子有组织架构、文件系统、语法树、决策树。

网格问题。解涉及空间位置。例子有路径规划、图像处理、棋盘问题。

识别建模对象是建模的第一步。如果你发现问题是关于"顺序"的，可能要用排列相关的算法。如果是关于"选择"的，可能要用子集或背包相关的算法。

---

## 建模的关键问题

建模时需要问几个关键问题。

输入有哪些？不只是数据，还有约束条件、目标函数、边界情况。

输出是什么？一个值？一个序列？一个结构？

目标是什么？最大化什么？最小化什么？判断是否存在？

约束有哪些？必须满足什么条件？哪些是硬约束，哪些是软约束？

这些问题看起来简单，但很多人想不清楚。他们直接跳到"用什么算法"，忽略了更基本的"我在解决什么问题"。

---

## 建模失误的代价

建模失误比算法失误代价更高。

如果你把问题建模错了，最优的算法解的也是错的问题。你会花很多时间优化一个不相关的目标，或者处理一个不存在的约束。

举个例子。某公司要优化仓库的拣货路径，目标是最小化总行走距离。工程师把它建模成一个旅行商问题：把所有要拣的货物位置看作城市，找一条经过所有城市的最短路径。

但这个建模忽略了一个关键因素：拣货员可以双手各拿一件货物。正确的建模应该是：把货物分成若干组，每组可以一次取完，最小化总距离。这不再是旅行商问题，而是一个更复杂的优化问题。

原来的建模花了很大力气求解，结果却不实用。因为问题本身建错了。

---

## 当你让 agent 帮你解决问题

回到我们的核心场景。你让 coding agent 帮你解决问题。

你需要做的不是直接给 agent 一个模糊的描述，而是先自己把问题建模清楚。

明确输入输出。给出具体的数据格式和范围。

识别问题类型。这是搜索？优化？判定？构造？

选择建模对象。是排列？子集？图？树？

确定算法方向。需要排序？查找？图算法？动态规划？

当你把这些想清楚了，再告诉 agent："这是一个图的最短路径问题，请用 Dijkstra 算法实现。"

这样的描述，agent 才能给你想要的代码。

---

## 小结

建模是人的核心能力。

它需要你理解问题的本质，识别问题的结构，选择合适的建模对象。这些判断 agent 做不了，因为它不知道你在解决什么问题，只知道你给了它什么描述。

建模失误的代价很高。如果问题建错了，最优的算法解的也是错的问题。

在让 agent 写代码之前，先自己把问题想清楚。这是节省时间的最好方式。
