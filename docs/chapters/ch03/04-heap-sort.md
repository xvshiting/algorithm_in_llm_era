- 区分堆排序与其他排序的特点，判断何时该用堆排序

## War Story：为什么选择堆排序？

2015年，某嵌入式系统团队接到一个棘手的需求：在内存只有 2KB 的微控制器上实现排序功能。

团队有两个备选方案：

- **快排**：期望很快，但最坏 Θ(n²)。万一遇到精心构造的输入，系统可能超时崩溃
- **归并排序**：稳定 Θ(n lg n)，但需要 Θ(n) 额外空间。2KB 内存撑不住

工程师小王皱着眉头说："快排太冒险，归并太费空间。就没有两全其美的方案吗？"

这时，资深工程师老张走进来，指着白板上的一个问题：**原地**和**最坏保证**能同时满足吗？

答案就在堆排序。

堆排序有三个独特优势：

1. **原地排序**：只需要 O(1) 额外空间（交换变量）
2. **最坏保证**：Θ(n lg n) 最坏情况，没有快排的 Θ(n²) 风险
3. **内存友好**：数据访问模式对缓存不友好（这是缺点，但在受限环境中可以接受）

最终，团队选择了堆排序。系统稳定上线，运行了三年没有问题。

**这个故事教我们什么？**

算法选择从来不是"哪个更快"的问题，而是"哪个权衡更适合当前场景"的问题。堆排序在"原地 + 最坏保证"这个组合上独一无二。如果你需要这两个性质同时满足，堆排序可能是唯一选择。

## 堆到底是什么？

在谈堆排序之前，让我们先理解"堆"这个数据结构。

**堆**是一个数组，但可以看作一棵完全二叉树。它满足两个性质：

### 1. 结构性质：完全二叉树

完全二叉树的意思是：除最后一层外，每层都是满的；最后一层从左到右填充。

让我们来看一个具体例子：

```
数组：[16, 14, 10, 8, 7, 9, 3, 2, 4]

树形表示：
             16           ← 第 0 层：1 个节点
           /    \
         14      10        ← 第 1 层：2 个节点
        /  \    /  \
       8    7  9    3      ← 第 2 层：4 个节点
      / \
     2   4                 ← 第 3 层：2 个节点（从左填充）

索引： 0   1   2  3  4  5  6  7  8
```

**为什么用数组表示树？**

因为完全二叉树的结构非常规律，我们可以用数学公式计算父子关系：

```text
父节点索引：parent(i) = (i - 1) // 2
左子节点索引：left(i) = 2 × i + 1
右子节点索引：right(i) = 2 × i + 2
```

你想想：如果用指针表示树，每个节点需要两个指针（左右孩子），还有指针本身的开销。数组表示法把所有节点"压缩"进连续内存，既省空间又省指针开销。

### 2. 堆序性质：父节点 ≥ 子节点

最大堆要求：每个节点的值 ≥ 它的所有子节点。

```
正确示例：
        16
       /  \
      14   10     ← 14 和 10 都 ≤ 16 ✓
     / \   / \
    8  7  9  3    ← 8, 7 ≤ 14; 9, 3 ≤ 10 ✓

错误示例：
        10
       /  \
      14   16     ← 14 和 16 都 > 10 ✗
```

**你想想**：为什么堆序性质只要求"父 ≥ 子"，不要求"左 > 右"或"完全有序"？

因为堆只关心一件事：**快速找到最大值**。堆序性质保证最大值在根节点（索引 0）。至于左右孩子的相对大小，不需要维护。这种"最小必要约束"的思想，是高效数据结构设计的关键。

## 堆的核心操作

堆有两个核心操作：**维护堆序性质**和**建堆**。理解这两个操作，就理解了堆排序的精髓。

### MAX-HEAPIFY：维护堆序性质

假设一棵树，根节点违反了堆序性质（比子节点小），但两棵子树都是合法的堆。怎么修复？

```
问题：
        4              ← 违反：4 < 14 和 4 < 10
       / \
      14  10           ← 子树都是合法堆
     / \  / \
    8  7 9  3

解决：让 4 "下沉"到正确位置

Step 1：比较 4 和子节点 14、10，14 最大
        交换 4 和 14
        
        14
       /  \
      4    10         ← 4 下沉到位置 1
     / \   / \
    8  7  9  3

Step 2：位置 1 的子树可能违反堆序性质
        比较 4 和子节点 8、7，8 最大
        交换 4 和 8

        14
       /  \
      8    10
     / \   / \
    4  7  9  3        ← 4 下沉到位置 3

Step 3：位置 3 是叶子节点，无需继续

结果：合法的最大堆
```

**伪代码**：

```python
MAX-HEAPIFY(A, i):
    l = left(i)      # 左子节点索引
    r = right(i)     # 右子节点索引
    largest = i      # 假设当前节点最大
    
    # 找出 i 和子节点中的最大值
    if l < len(A) and A[l] > A[largest]:
        largest = l
    if r < len(A) and A[r] > A[largest]:
        largest = r
    
    # 如果最大值不是当前节点，交换并递归
    if largest != i:
        swap A[i] and A[largest]
        MAX-HEAPIFY(A, largest)
```

**复杂度分析**：

每次递归，节点高度减 1。树高度为 h 的节点，最多递归 h 次。每次递归是 Θ(1)。所以 MAX-HEAPIFY 的复杂度是 Θ(h)。

对于 n 个节点的堆，高度最多 ⌊lg n⌋。所以 MAX-HEAPIFY 的最坏复杂度是 Θ(lg n)。

### BUILD-MAX-HEAP：从乱序数组建堆

现在问题来了：给定一个乱序数组，怎么把它变成堆？

**朴素想法**：逐个插入，每次调用 MAX-HEAPIFY。但这样是 Θ(n lg n)。

**更聪明的方法**：从最后一个非叶子节点开始，自底向上调用 MAX-HEAPIFY。

```
BUILD-MAX-HEAP(A):
    for i = len(A)//2 down to 0:
        MAX-HEAPIFY(A, i)
```

**为什么从 len(A)//2 开始？**

因为索引 ≥ len(A)//2 的节点都是叶子节点，天然满足堆序性质。只有非叶子节点需要处理。

让我们手动建堆：

```
输入：[4, 1, 3, 2, 16, 9, 10, 14, 8, 7]

树形表示：
         4
      /     \
     1       3
    / \     / \
   2   16  9   10
  / \ / \
 14 8 7

非叶子节点索引：4, 3, 2, 1, 0（共 5 个）

i=4：对节点 16 做 MAX-HEAPIFY
      16 有子节点 7，但 16 > 7，无需交换
      
i=3：对节点 2 做 MAX-HEAPIFY
      2 有子节点 14 和 8，14 最大，交换
      得到：[4, 1, 3, 14, 16, 9, 10, 2, 8, 7]

i=2：对节点 3 做 MAX-HEAPIFY
      3 有子节点 9 和 10，10 最大，交换
      得到：[4, 1, 10, 14, 16, 9, 3, 2, 8, 7]

i=1：对节点 1 做 MAX-HEAPIFY
      1 有子节点 14 和 16，16 最大，交换
      得到：[4, 16, 10, 14, 1, 9, 3, 2, 8, 7]
      1 下沉到位置 4，有子节点 8 和 7
      1 < 8，交换
      得到：[4, 16, 10, 14, 8, 9, 3, 2, 1, 7]

i=0：对节点 4 做 MAX-HEAPIFY
      4 有子节点 16 和 10，16 最大，交换
      得到：[16, 4, 10, 14, 8, 9, 3, 2, 1, 7]
      4 下沉到位置 1，有子节点 14 和 8
      4 < 14，交换
      得到：[16, 14, 10, 4, 8, 9, 3, 2, 1, 7]
      4 下沉到位置 3，有子节点 2 和 1
      4 > 2 且 4 > 1，无需继续

最终堆：[16, 14, 10, 4, 8, 9, 3, 2, 1, 7]
```

## 建堆为什么是线性时间？

这里有一个反直觉的结论：**BUILD-MAX-HEAP 的时间复杂度是 Θ(n)，不是 Θ(n lg n)**。

很多初学者会这样分析：

```
MAX-HEAPIFY 是 Θ(lg n)
调用 n/2 次
总时间 = Θ(n lg n)  ← 这个分析是错的！
```

**错在哪里？**

错在假设每次 MAX-HEAPIFY 都是 Θ(lg n)。实际上，大部分节点的高度很小！

让我们精确计算：

```
高度为 h 的节点数量 ≤ ⌈n / 2^{h+1}⌉

为什么？因为：
- 高度 0（叶子节点）：约 n/2 个
- 高度 1：约 n/4 个
- 高度 2：约 n/8 个
- ...
- 高度 lg n：最多 1 个（根节点）
```

**精确计算**：

```
总时间 = Σ_{h=0}^{⌊lg n⌋} (高度 h 的节点数) × Θ(h)
       ≤ Σ_{h=0}^{⌊lg n⌋} ⌈n/2^{h+1}⌉ × c×h
       ≤ c×n × Σ_{h=0}^{∞} h / 2^{h+1}
       = c×n × 2        （数学级数求和）
       = O(n)
```

**直觉理解**：

- 大部分节点（叶子节点）高度为 0，处理它们是 Θ(1)
- 高度大的节点很少（只有根节点高度最大）
- 所以总时间不是 n × lg n，而是"数量 × 高度"的加权和

这是算法分析中的经典技巧：**不要用最坏情况乘以次数，要分析不同层次的开销**。

## 堆排序：完整流程

现在我们把所有操作串起来：

```
HEAPSORT(A):
    BUILD-MAX-HEAP(A)           // Θ(n)
    for i = len(A)-1 down to 1:  // n-1 次循环
        swap A[0] and A[i]      // 最大元素放到末尾
        MAX-HEAPIFY(A[0..i-1], 0)  // 维护堆性质 Θ(lg i)
```

**图解说明**：

```
初始数组：[4, 1, 3, 2, 16, 9, 10, 14, 8, 7]

Step 1：建堆
得到：[16, 14, 10, 4, 8, 9, 3, 2, 1, 7]

Step 2：循环排序

i=9: swap A[0]和A[9] → [7, 14, 10, 4, 8, 9, 3, 2, 1 | 16]
     heapify → [14, 8, 10, 4, 7, 9, 3, 2, 1 | 16]

i=8: swap → [1, 8, 10, 4, 7, 9, 3, 2 | 14, 16]
     heapify → [10, 8, 9, 4, 7, 1, 3, 2 | 14, 16]

i=7: swap → [2, 8, 9, 4, 7, 1, 3 | 10, 14, 16]
     heapify → [9, 8, 3, 4, 7, 1, 2 | 10, 14, 16]

i=6: swap → [2, 8, 3, 4, 7, 1 | 9, 10, 14, 16]
     heapify → [8, 7, 3, 4, 2, 1 | 9, 10, 14, 16]

i=5: swap → [1, 7, 3, 4, 2 | 8, 9, 10, 14, 16]
     heapify → [7, 4, 3, 1, 2 | 8, 9, 10, 14, 16]

i=4: swap → [2, 4, 3, 1 | 7, 8, 9, 10, 14, 16]
     heapify → [4, 2, 3, 1 | 7, 8, 9, 10, 14, 16]

i=3: swap → [1, 2, 3 | 4, 7, 8, 9, 10, 14, 16]
     heapify → [3, 2, 1 | 4, 7, 8, 9, 10, 14, 16]

i=2: swap → [1, 2 | 3, 4, 7, 8, 9, 10, 14, 16]
     heapify → [2, 1 | 3, 4, 7, 8, 9, 10, 14, 16]

i=1: swap → [1 | 2, 3, 4, 7, 8, 9, 10, 14, 16]
     heapify → [1 | 2, 3, 4, 7, 8, 9, 10, 14, 16]

最终结果：[1, 2, 3, 4, 7, 8, 9, 10, 14, 16]  ✓
```

**复杂度**：

```
建堆：Θ(n)
循环：Σ_{i=1}^{n-1} Θ(lg i) = Θ(n lg n)

总复杂度：Θ(n lg n)
```

## 堆排序 vs 快排 vs 归并排序

让我们对比三种 O(n lg n) 排序算法：

| 特性 | 堆排序 | 快排 | 归并排序 |
|------|--------|------|----------|
| **最坏时间** | Θ(n lg n) | Θ(n²) | Θ(n lg n) |
| **期望时间** | Θ(n lg n) | Θ(n lg n) | Θ(n lg n) |
| **额外空间** | Θ(1) | Θ(lg n) 递归栈 | Θ(n) 额外数组 |
| **稳定性** | 不稳定 | 不稳定 | 稳定 |
| **缓存友好** | 不友好 | 友好 | 中等 |
| **适用场景** | 内存受限、需要最坏保证 | 一般场景 | 大数据、需要稳定 |

**选择指南**：

- **选堆排序**：内存紧张 + 不能接受最坏 Θ(n²)
- **选快排**：一般场景，实践中最快（缓存友好 + 常数小）
- **选归并**：数据量大 + 需要稳定性 + 有额外空间

**一个有趣的对比**：

快排的 Θ(n²) 最坏情况在实践中很少遇到，但理论上存在。在某些关键场景（如实时系统、嵌入式设备），不能接受"概率性快"——必须保证"最坏也够快"。这时堆排序的价值就体现出来了。

## 与 LLM 的关联

堆排序的思想在 LLM 中也有体现：

### 1. Top-k 采样

LLM 生成时需要从概率分布中选 Top-k 个 token。朴素方法是排序后取前 k，Θ(n lg n)。

更聪明的方法是用堆：维护一个大小为 k 的最小堆，Θ(n lg k)。

```
朴素方法：排序 → 取前 k → Θ(n lg n)
堆方法：维护 k 大小的堆 → Θ(n lg k)

当 k << n 时，堆方法快很多
```

### 2. 优先队列

LLM 推理时的 beam search 需要维护候选序列。堆可以作为优先队列，高效获取和更新 Top-k。

## 本节要点

让我们回顾这一节学到了什么：

1. **堆的本质**：完全二叉树 + 堆序性质，用数组紧凑表示
2. **核心操作**：MAX-HEAPIFY 让节点"下沉"到正确位置，Θ(lg n)
3. **反直觉结论**：建堆是 Θ(n)，不是 Θ(n lg n)。关键在于大部分节点高度很小
4. **堆排序流程**：建堆 → 交换堆顶和末尾 → heapify → 重复
5. **适用场景**：原地 + 最坏保证的组合，堆排序是独特选择

**与后续内容的衔接**：

- 堆不仅是排序工具，更是优先队列的基础
- 下一节我们学习线性时间排序（计数排序、基数排序），它们利用"数据范围小"的特殊结构
- 堆排序是通用排序，线性时间排序是特殊场景排序——这体现了"通用 vs 专用"的设计权衡
