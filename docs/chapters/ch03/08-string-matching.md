## 问题情境

某搜索团队发现搜索延迟从 200ms 上升到 500ms，用户投诉激增。

排查发现：朴素字符串匹配太慢。

```
文档库：100 万篇，平均 5000 字符
关键词：平均 5 字符

朴素匹配：5000 × 5 × 100万 = 250 亿比较

太慢！
```

改用 Rabin-Karp + Bloom Filter 预过滤，延迟降到 50ms。

**问题**：为什么 Rabin-Karp 更快？

---

## 直观思路

**字符串匹配问题**：文本 T（长度 n），模式 P（长度 m），找出所有匹配位置。

**朴素方法**：逐位置尝试，每次 O(m) 比较 → O(nm)

**三种优化思路**：

1. **Rabin-Karp**：用哈希比较代替逐字符比较
2. **KMP**：失败后"跳过"已匹配部分
3. **Boyer-Moore**：从右往左匹配，跳过更多

---

## 规范定义

### 字符串匹配

**输入**：文本 T（长度 n），模式 P（长度 m）
**输出**：所有 P 在 T 中出现的位置

### 复杂度对比

| 算法 | 预处理 | 匹配时间 | 特点 |
|------|--------|---------|------|
| 朴素 | 无 | O(nm) | 简单 |
| Rabin-Karp | O(m) | O(n) 期望 | 哈希 |
| KMP | O(m) | O(n) | 失败函数 |

---

## 算法实现

### Rabin-Karp：滚动哈希

```python
def rabin_karp(T, P):
    n, m = len(T), len(P)
    p_hash = hash(P)              # 模式哈希
    
    # 滚动哈希
    t_hash = hash(T[0:m])
    for s in range(n - m + 1):
        if t_hash == p_hash:
            if T[s:s+m] == P:      # 验证（避免假阳性）
                return s
        t_hash = roll(t_hash, T[s], T[s+m])  # O(1)
```

**滚动哈希数学**：

把字符串看作 d 进制数：

```
H = Σ c_i × d^{m-1-i}

从 H(T[s..s+m-1]) 到 H(T[s+1..s+m])：
去掉 T[s]，加入 T[s+m]
= (H - T[s] × d^{m-1}) × d + T[s+m]
```

### KMP：失败函数

```python
def kmp(T, P):
    # 预处理：计算失败函数
    failure = compute_failure(P)
    
    j = 0  # 模式位置
    for i in range(len(T)):
        while j > 0 and T[i] != P[j]:
            j = failure[j - 1]     # 跳到已匹配部分
        if T[i] == P[j]:
            j += 1
        if j == len(P):
            return i - len(P) + 1
```

**失败函数**：模式串的"自匹配"

```
P = "ABABAC"
failure = [0, 0, 1, 2, 3, 0]

含义：
- P[0..1] = "AB"，无前缀后缀重叠 → failure[1] = 0
- P[0..2] = "ABA"，"A"重叠 → failure[2] = 1
- P[0..3] = "ABAB"，"AB"重叠 → failure[3] = 2
```

---

## 这个方法是怎么想到的

### Rabin-Karp 的思路

**问题**：逐字符比较太慢。

**洞察**：哈希比较更快。相等 → 可能匹配 → 验证。

**关键**：滚动哈希，O(1) 更新。

### KMP 的思路

**问题**：失败后从头开始，浪费已匹配信息。

**洞察**：模式串可以"记忆"自己的结构。

**关键**：失败函数告诉我们要跳到哪里。

---

## 正确性分析

### Rabin-Karp：蒙特卡罗验证

哈希相等 → 可能匹配（假阳性）→ 验证排除。

假阳性概率 = 1/质数（哈希函数设计）。

### KMP：失败函数保证

失败函数保证不会漏掉匹配位置。

跳到失败位置后，前面的部分已经匹配，不需要重新比较。

---

## 复杂度分析

| 算法 | 时间 | 特点 |
|------|------|------|
| 朴素 | O(nm) | 简单，最坏 |
| Rabin-Karp | O(n) 期望 | 哈希，概率 |
| KMP | O(n) | 最坏保证 |

---

## 什么时候不该用这个方法

### 该用朴素的情况

- 模式很短（m 很小）
- 匹配很少（不匹配后快速失败）

### 该用 Rabin-Karp 的情况

- 多模式匹配（多个哈希同时比较）
- 可以接受概率性结果

### 该用 KMP 的情况

- 需要最坏保证
- 模式有重复结构

---

## 本节小结

**解决了什么问题？**

快速字符串匹配，O(n) 而不是 O(nm)。

**核心方法是什么？**

- **Rabin-Karp**：滚动哈希 + 蒙特卡罗验证
- **KMP**：失败函数 + 自匹配洞察

**为什么正确？**

哈希相等概率保证。失败函数跳过已匹配部分。

**代价是什么？**

Rabin-Karp：假阳性需验证。KMP：预处理 O(m)。

**适用场景？**

文本检索、多模式匹配。
