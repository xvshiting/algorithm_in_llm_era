## War Story：为什么搜索引擎需要快速字符串匹配？

2023年，某搜索团队优化全文检索。工程师发现，用朴素字符串匹配检查每个文档是否包含关键词，O(nm) 太慢。

团队改用 Rabin-Karp，期望 O(n+m)，配合 Bloom Filter 预过滤，搜索延迟从 500ms 降到 50ms。

**教训**：字符串匹配在文本检索中是核心操作，选择合适的算法影响用户体验。

---

## 问题定义

**输入**：文本 T（长度 n），模式 P（长度 m）

**输出**：所有 P 在 T 中出现的位置

**朴素方法**：O(nm)

```
NAIVE-STRING-MATCH(T, P):
    for s = 0 to n-m:
        if T[s..s+m-1] == P:
            输出 s
```

每次检查需要 O(m) 比较，总共 O(nm)。

---

## Rabin-Karp：滚动哈希

**核心思想**：用哈希比较代替逐字符比较

**滚动哈希**：

如果 H(P) 和 H(T[s..s+m-1]) 相等，再逐字符验证（避免哈希冲突）。

关键：从 H(T[s..s+m-1]) 计算 H(T[s+1..s+m]) 只需要 O(1)

**算法**：

```
RABIN-KARP(T, P, d, q):
    // d 是字符基数，q 是哈希模数
    n = len(T), m = len(P)
    h = d^{m-1} mod q  // 最高位权重
    
    // 计算 P 的哈希和 T[0..m-1] 的哈希
    p = 0, t = 0
    for i = 0 to m-1:
        p = (d × p + P[i]) mod q
        t = (d × t + T[i]) mod q
    
    for s = 0 to n-m:
        if p == t:  // 哈希相等
            if T[s..s+m-1] == P:  // 验证（避免假阳性）
                输出 s
        if s < n-m:  // 滚动计算下一个哈希
            t = (d × (t - T[s] × h) + T[s+m]) mod q
```

**复杂度**：

- 预处理：Θ(m)
- 匹配：期望 Θ(n + m)，最坏 Θ(nm)（大量假阳性）

**蒙特卡罗视角**：

如果 q 选择足够大（如 q >> n），假阳性概率可以控制：

```
P[哈希冲突] ≤ n/q  // n 次匹配尝试，每次冲突概率 1/q
```

选择 q ≈ n^2，假阳性概率 ≤ 1/n，期望验证次数 ≤ 1。

## KMP：失败函数

**核心思想**：利用模式串对自身的匹配信息，避免文本回溯

**失败函数** π：

```
π[i] = P[0..i-1] 的最长真前缀同时也是后缀的长度
```

**图解说明**：

```
模式：P = "ababab"
π: [-1, 0, 0, 1, 2, 3]

P[0..2] = "aba"
最长真前缀同时也是后缀 = "a"（长度 1）
所以 π[3] = 1

P[0..4] = "ababa"
最长真前缀同时也是后缀 = "aba"（长度 3）
所以 π[5] = 3
```

**为什么有用？**

假设在位置 i 匹配失败（T[j] ≠ P[i]），我们不需要从头开始，而是跳到 π[i]：

- P[0..π[i]-1] 已经匹配 T[j-π[i]..j-1]
- 从位置 π[i] 继续

**算法**：

```
KMP-MATCHER(T, P):
    // 预处理：计算失败函数
    π = COMPUTE-PI(P)
    
    q = 0  // 已匹配的模式字符数
    for i = 0 to n-1:
        while q > 0 and P[q] != T[i]:
            q = π[q]  // 利用失败信息回溯
        if P[q] == T[i]:
            q++
        if q == m:  // 完全匹配
            输出 i - m + 1
            q = π[q]
```

**复杂度**：Θ(n + m) 最坏保证

**核心洞察**：模式串对自身的匹配提供了"跳过不可能位置"的信息。

## 与 LLM 的关联

### 1. Tokenization（BPE）

BPE（Byte Pair Encoding）本质是字符串匹配 + 合并：

```
输入："lower"
词汇表：{"l", "o", "w", "e", "r", "lo", "low", "er", "lower"}

Step 1：匹配最长子串 "low" → ["low", "e", "r"]
Step 2：匹配最长子串 "er" → ["low", "er"]
Step 3：匹配最长子串 "lower" → ["lower"]

输出：["lower"]（单个 token）
```

核心操作：在文本中找词汇表中最长的匹配子串。

### 2. RAG 检索

RAG（Retrieval-Augmented Generation）的检索阶段：

```
输入：用户查询 Q
输出：相关文档 D[1..k]

检索方法：
- BM25：基于词频的字符串匹配扩展
- Embedding 搜索：向量空间中的"软匹配"
- 精确匹配：字符串匹配 + 正则表达式
```

字符串匹配是 RAG 检索的基础操作。

### 3. LLM 输出验证

Rabin-Karp 的蒙特卡罗验证可以类比 LLM 输出验证：

```
Rabin-Karp：
- 哈希相等 → 可能匹配（假阳性）
- 逐字符验证 → 确认匹配

LLM 采样：
- 采样输出 → 可能正确（假阳性）
- 人工/自动验证 → 确认正确
```

两者都用"概率性快速检测 + 确定性验证"模式。

## 算法选择指南

| 场景 | 推荐算法 |
|------|----------|
| 单模式，需要最坏保证 | KMP |
| 多模式匹配 | Aho-Corasick |
| 需要概率性快速检测 | Rabin-Karp |
| 正则表达式匹配 | DFA/NFA |
| 大规模文本检索 | BM25 + 倒排索引 |
