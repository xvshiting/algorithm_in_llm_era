## War Story：搜索引擎的生死线

2023年，某搜索团队优化全文检索。工程师小张发现，搜索延迟从 200ms 上升到 500ms，用户投诉量激增。

排查后发现：朴素字符串匹配检查每个文档是否包含关键词，O(nm)，太慢。

```
文档库：100 万篇文档，平均长度 5000 字符
关键词：平均长度 5 字符

朴素匹配：
每篇文档：5000 × 5 = 25000 次字符比较
100 万篇：25000 × 100万 = 250 亿次比较

这太慢了！
```

小张改用 Rabin-Karp：期望 O(n+m)，配合 Bloom Filter 预过滤（快速排除不包含关键词的文档），搜索延迟从 500ms 降到 50ms。

**关键决策**：

1. **预过滤**：用 Bloom Filter 快速判断文档"可能包含关键词"
2. **Rabin-Karp**：快速验证是否真的包含
3. **蒙特卡罗验证**：哈希相等 → 可能匹配 → 验证（避免假阳性）

**这个故事教我们什么？**

字符串匹配在文本检索中是核心操作。选择合适的算法影响用户体验——50ms vs 500ms，用户感知完全不同。

更重要的是，Rabin-Karp 的设计哲学（概率换取效率）与 Bloom Filter 的设计哲学（假阳性可接受）相结合，形成了"快速预过滤 + 精确验证"的两阶段流程。这个模式在搜索引擎、LLM 采样、RAG 检索中都有体现。

## 问题定义：字符串匹配到底是什么？

**输入**：文本 T（长度 n），模式 P（长度 m）

**输出**：所有 P 在 T 中出现的位置

**让我们看一个具体例子**：

```
文本 T = "ABABCABCABAB"
模式 P = "ABAB"

期望输出：位置 0 和 8

T: A B A B C A B C A B A B
P: A B A B                 ← 位置 0 匹配 ✓
      C ...                ← 位置 4 不匹配
                A B A B    ← 位置 8 匹配 ✓
```

**朴素方法**：O(nm)

```
NAIVE-STRING-MATCH(T, P):
    for s = 0 to n-m:            # n-m+1 次尝试
        if T[s..s+m-1] == P:     # 每次尝试最多 m 次比较
            输出 s
```

每次检查需要 O(m) 比较，总共 O(nm)。

**你想想**：为什么朴素方法这么慢？

因为每次失败后，我们从下一个位置重新开始，完全不利用之前的匹配信息。如果 T[s..s+m-1] 和 P 大部分匹配，只最后一位不匹配，朴素方法不会"记住"已经匹配的部分。

---

## Rabin-Karp：用哈希快速比较

**核心思想**：用哈希比较代替逐字符比较

如果 H(P) 和 H(T[s..s+m-1]) 相等，再逐字符验证（避免哈希冲突）。

关键：从 H(T[s..s+m-1]) 计算 H(T[s+1..s+m]) 只需要 O(1)——这就是"滚动哈希"。

### 滚动哈希的数学

把字符串看作数字（d进制）：

```
字符串 "abc" 的哈希 = a × d^2 + b × d + c

从 T[s..s+m-1] 到 T[s+1..s+m]：
去掉最高位 T[s]，加入新最低位 T[s+m]

H(T[s+1..s+m]) = (H(T[s..s+m-1]) - T[s] × d^{m-1}) × d + T[s+m]
              = d × H(T[s..s+m-1]) - T[s] × d^m + T[s+m]

用模 q 避免溢出：
H(T[s+1..s+m]) mod q = (d × (H mod q - T[s] × h) + T[s+m]) mod q
其中 h = d^{m-1} mod q
```

### 算法实现

```
RABIN-KARP(T, P, d, q):
    n = len(T), m = len(P)
    h = d^{m-1} mod q
    
    # 预处理：计算 P 的哈希和 T[0..m-1] 的哈希
    p = 0, t = 0
    for i = 0 to m-1:
        p = (d × p + P[i]) mod q
        t = (d × t + T[i]) mod q
    
    for s = 0 to n-m:
        if p == t:                     # 哈希相等
            if T[s..s+m-1] == P:       # 验证（避免假阳性）
                输出 s
        if s < n-m:                    # 滚动计算下一个哈希
            t = (d × (t - T[s] × h) + T[s+m]) mod q
```

### 具体例子

```
T = "235902671"，P = "267"
d = 10（十进制），q = 13（模数）

预处理：
h = 10^{2} mod 13 = 100 mod 13 = 9

P = "267"：
p = (10 × 0 + 2) mod 13 = 2
p = (10 × 2 + 6) mod 13 = 26 mod 13 = 0
p = (10 × 0 + 7) mod 13 = 7

T[0..2] = "235"：
t = (10 × 0 + 2) mod 13 = 2
t = (10 × 2 + 3) mod 13 = 23 mod 13 = 10
t = (10 × 10 + 5) mod 13 = 105 mod 13 = 1

匹配：
s=0: p=7, t=1，不相等
s=1: 滚动哈希
    t = (10 × (1 - T[0]×h) + T[3]) mod 13
    t = (10 × (1 - 2×9) + 9) mod 13
    t = (10 × (1 - 18) + 9) mod 13
    t = (10 × (-17) + 9) mod 13 = -170 + 9 mod 13
    t = -161 mod 13 = 13 - 161 mod 13 ... 
    
    简化：t = (10 × (1 - 18) + 9) mod 13 = (-170 + 9) mod 13
    = -161 mod 13 = 13 × 13 - 161 = 169 - 161 = 8
    
p=7, t=8，不相等

...继续...

找到哈希相等的位置后，逐字符验证。
```

### 蒙特卡罗视角

如果 q 选择足够大（如 q >> n），假阳性概率可以控制：

```
P[哈希冲突] ≤ n/q  # n 次匹配尝试，每次冲突概率 1/q

选择 q ≈ n^2，假阳性概率 ≤ 1/n，期望验证次数 ≤ 1
```

**蒙特卡罗 vs 拉斯维加斯**：

- **蒙特卡罗算法**：可能出错（假阳性），但时间保证
- **拉斯维加斯算法**：结果正确（验证后），但时间期望

Rabin-Karp 是两者结合：哈希阶段是蒙特卡罗（可能假阳性），验证阶段是拉斯维加斯（保证正确）。

**复杂度**：

- 预处理：Θ(m)
- 匹配：期望 Θ(n + m)，最坏 Θ(nm)（大量假阳性）

## KMP：利用模式串的自匹配信息

**核心思想**：利用模式串对自身的匹配信息，避免文本回溯

朴素方法的问题：匹配失败后，从下一个位置重新开始，浪费了已经匹配的信息。

KMP 的解决方案：**失败函数**——模式串"记忆"自己的结构。

### 失败函数 π

```
π[i] = P[0..i-1] 的最长真前缀同时也是后缀的长度
```

**让我们看一个具体例子**：

```
模式 P = "ABABAB"

计算 π：
i=0: P[0..-1] = ""（空），π[0] = -1（约定）
i=1: P[0..0] = "A"，最长真前缀也是后缀？没有，π[1] = 0
i=2: P[0..1] = "AB"，最长真前缀也是后缀？没有，π[2] = 0
i=3: P[0..2] = "ABA"，最长真前缀也是后缀？"A"，π[3] = 1
i=4: P[0..3] = "ABAB"，最长真前缀也是后缀？"AB"，π[4] = 2
i=5: P[0..4] = "ABABA"，最长真前缀也是后缀？"ABA"，π[5] = 3

π = [-1, 0, 0, 1, 2, 3]
```

**为什么有用？**

假设在位置 i 匹配失败（T[j] ≠ P[i]），我们不需要从头开始，而是跳到 π[i]：

- P[0..π[i]-1] 已经匹配 T[j-π[i]..j-1]
- 从位置 π[i] 继续

**直觉理解**：

模式串"ABA"的前缀"A"同时也是后缀"A"。如果我们在位置 3 匹配失败，但前面的"ABA"已经匹配，我们可以利用这个信息——"ABA"的后缀"A"可以继续匹配新的位置。

### 算法实现

```
COMPUTE-PI(P):
    m = len(P)
    π = [-1] × m
    k = -1
    
    for i = 1 to m-1:
        while k >= 0 and P[k+1] != P[i]:
            k = π[k]
        if P[k+1] == P[i]:
            k++
        π[i] = k
    
    return π

KMP-MATCHER(T, P):
    n = len(T), m = len(P)
    π = COMPUTE-PI(P)
    
    q = 0  # 已匹配的模式字符数
    for i = 0 to n-1:
        while q > 0 and P[q] != T[i]:
            q = π[q]  # 利用失败信息
        if P[q] == T[i]:
            q++
        if q == m:  # 完全匹配
            输出 i - m + 1
            q = π[q]
```

### 具体例子

```
T = "ABABCABAB"，P = "ABABAB"，π = [-1, 0, 0, 1, 2, 3]

执行过程：

i=0, T[0]="A", q=0, P[0]="A", 匹配，q=1
i=1, T[1]="B", q=1, P[1]="B", 匹配，q=2
i=2, T[2]="A", q=2, P[2]="A", 匹配，q=3
i=3, T[3]="B", q=3, P[3]="B", 匹配，q=4
i=4, T[4]="C", q=4, P[4]="A", 不匹配
    while q>0 and P[q]!=T[i]: q = π[q]
    q = π[4] = 2
    P[2]="A", T[4]="C", 不匹配
    q = π[2] = 0
    P[0]="A", T[4]="C", 不匹配
    q = 0
i=5, T[5]="A", q=0, P[0]="A", 匹配，q=1
...

关键：失败后不需要从 T[4] 开始重新匹配，而是利用 π 跳过不可能的位置
```

**复杂度**：Θ(n + m) 最坏保证

**核心洞察**：模式串对自身的匹配提供了"跳过不可能位置"的信息。这是"自我认知"——模式串"知道"自己的结构，可以利用这个知识优化匹配。

## 与 LLM 的关联

字符串匹配的思想在 LLM 中也有体现：

### 1. Tokenization（BPE）：最长匹配

BPE（Byte Pair Encoding）本质是字符串匹配 + 合并：

```
输入："lower"
词汇表：{"l", "o", "w", "e", "r", "lo", "low", "er", "lower"}

Step 1：匹配最长子串 "low" → ["low", "e", "r"]
Step 2：匹配最长子串 "er" → ["low", "er"]
Step 3：匹配最长子串 "lower" → ["lower"]

输出：["lower"]（单个 token）
```

核心操作：在文本中找词汇表中最长的匹配子串。

这是字符串匹配的变体——不是找单个模式，而是找词汇表中最长的匹配。

### 2. RAG 检索：软匹配 vs 硬匹配

RAG（Retrieval-Augmented Generation）的检索阶段：

```
输入：用户查询 Q
输出：相关文档 D[1..k]

检索方法：
- BM25：基于词频的"软匹配"（相似度排序）
- Embedding 搜索：向量空间中的"软匹配"
- 精确匹配：字符串匹配 + 正则表达式（"硬匹配")
```

**软匹配 vs 硬匹配**：

- **硬匹配**（字符串匹配）：精确匹配，返回包含关键词的文档
- **软匹配**（Embedding）：语义匹配，返回"语义相似"的文档（即使不包含关键词）

LLM 的 Embedding 匹配类似 Rabin-Karp 的"哈希匹配"——不是精确字符匹配，而是"语义哈希"匹配。

### 3. 输出验证：概率检测 + 确定性验证

Rabin-Karp 的蒙特卡罗验证类比 LLM 输出验证：

```
Rabin-Karp：
- 哈希相等 → 可能匹配（假阳性）
- 逐字符验证 → 确认匹配（确定性）

LLM 采样：
- 采样输出 → 可能正确（假阳性）
- 人工/自动验证 → 确认正确（确定性）

两者都是"概率性快速检测 + 确定性验证"模式
```

## 算法选择指南

让我们总结何时用哪种算法：

| 场景 | 推荐算法 | 理由 |
|------|----------|------|
| 单模式，需要最坏保证 | KMP | Θ(n+m) 最坏保证 |
| 多模式匹配 | Aho-Corasick | 自动机处理多模式 |
| 需要概率性快速检测 | Rabin-Karp | 期望 Θ(n+m)，可配合预过滤 |
| 正则表达式匹配 | DFA/NFA | 正则表达式的自动机 |
| 大规模文本检索 | BM25 + 倒排索引 | 索引 + 排序 |
| 实时流处理 | 滚动哈希 | O(1) 滚动更新 |

**选择决策树**：

```
需要最坏保证？
├─ 是 → KMP
└─ 否 → 需要概率性快速检测？
    ├─ 是 → Rabin-Karp + Bloom Filter
    └─ 否 → 多模式匹配？
        ├─ 是 → Aho-Corasick
        └─ 否 → 朴素方法（简单场景）
```

## 本节要点

让我们回顾这一节学到了什么：

1. **朴素匹配**：O(nm)，每次失败后不利用匹配信息
2. **Rabin-Karp**：滚动哈希，期望 Θ(n+m)，最坏 Θ(nm)。蒙特卡罗 + 拉斯维加斯结合
3. **KMP**：失败函数，利用模式串的自匹配信息，Θ(n+m) 最坏保证
4. **设计哲学**："概率换取效率"（Rabin-Karp）vs "自我认知优化"（KMP）
5. **LLM 关联**：Tokenization（最长匹配）、RAG（软匹配 vs 硬匹配）、验证（概率 + 确定性）

**与后续内容的衔接**：

- 下一节学习二分搜索，也是"利用结构信息优化查找"的例子
- 字符串匹配的"滚动哈希"思想在哈希（上一节）中体现
- KMP 的"失败函数"类似哈希的"预计算结构"——利用已知信息优化未知匹配
