## 层次一：基础练习

### E1. 决策树下界证明

**任务**：用决策树模型证明 3 个元素的排序至少需要 5 次比较（最坏情况）。

**提示**：
- 3 个元素有 3! = 6 种排列
- 高度 h 的二叉树最多有 2^h 个叶子
- 证明 h ≥ lg(6) ≈ 2.58，所以 h ≥ 3
- 但仔细想：高度 3 的树最多 8 个叶子，可以覆盖 6 种排列
- 为什么是 5 次而不是 3 次？

**答案要点**：
- 决策树必须是"完整"的——每个叶子对应一种排列
- 实际构造：高度 3 的树可以覆盖 6 种排列，但某些分支可能需要额外比较
- 最坏情况下确实需要 5 次比较（某些排列的路径长度）

### E2. 循环不变式证明

**任务**：用循环不变式证明 PARTITION 的正确性。

**循环不变式**：
> 在每次 for 循环迭代开始时：
> - A[p..i] 的元素都 ≤ pivot
> - A[i+1..j-1] 的元素都 > pivot
> - A[r] 是 pivot

**三步证明**：
1. 初始化（j = p）
2. 保持（每次迭代）
3. 终止（j = r）

### E3. 三种线性排序对比

**任务**：比较计数排序、基数排序、桶排序的适用条件和限制。

**场景判断**：

| 场景 | 推荐算法 | 原因 |
|------|----------|------|
| 年龄排序（0-150岁） | 计数排序 | 整数键，范围 k=150 << n |
| 电话号码排序 | 基数排序 | 11 位数字，每位范围 k=10 |
| 随机生成的成绩（0-100） | 桶排序或计数排序 | 均匀分布或范围有限 |
| 字符串按字典序排序 | 基数排序 | 字符为"位"，范围 k=26（小写字母） |
| 一般整数排序（范围未知） | 快排/堆排序 | k 可能太大 |

## 层次二：LLM 协同练习

### E4. McIlroy 杀手序列

**任务**：让 LLM 实现一个确定性快排（选第一个元素作为 pivot），构造 McIlroy 杀手序列使其退化为 Θ(n²)。

**执行步骤**：
1. 让 LLM 实现快排（pivot = A[0]）
2. 让 LLM 构造杀手序列
3. 运行验证

**审查要点**：
- LLM 是否理解"杀手序列"的原理？
- 是否构造出有效的退化输入？
- 能否解释为什么这个输入会导致 Θ(n²)？

**答案要点**：
- 杀手序列：每次 pivot 都是最小或最大元素
- 例如：递增序列 [1, 2, 3, ..., n] 如果 pivot = A[0]
- 每次分割：左边空，右边 n-1 个元素
- 总时间：Σ_{k=1}^{n} Θ(k) = Θ(n²)

### E5. 哈希策略对比分析

**任务**：给定一个应用场景（缓存系统），比较三种哈希策略（链式、开放寻址、Bloom 过滤器）的优劣。

**场景描述**：
- 1000 万条 URL 缓存
- 查询频率：每秒 10 万次
- 内存限制：100 MB
- 假阳性可接受（用户会重新请求），假阴性不可接受（缓存命中率下降）

**让 LLM 分析**，你评判：

| 策略 | 内存占用 | 查询时间 | 假阳性 | 假阴性 | 适用性 |
|------|----------|----------|--------|--------|--------|
| 链式哈希 | 高（存 URL） | O(1) 期望 | 0 | 0 | 内存不足 |
| 开放寻址 | 中（存 URL） | O(1) 期望 | 0 | 0 | 可能可行 |
| Bloom Filter | 低（只存哈希） | O(k) | 可控 | 0 | 最适用 |

**审查要点**：
- LLM 是否正确理解 Bloom Filter 的假阳性/假阴性特性？
- 是否给出合理的哈希函数数量 k？
- 是否考虑删除操作的限制？

### E6. 排序三元悖在 LLM 推理中的体现

**任务**：探讨排序三元悖在 LLM 推理中的类比。

**三元悖**：O(n) 时间 + 稳定 + 原地——三者只能取二

**LLM 推理类比**：

| 特性 | 含义 | LLM 实现 |
|------|------|----------|
| **O(1) 时间（单步）** | 每次生成 O(1) token | 自回归生成 ✓ |
| **稳定** | 相同输入产生相同输出 | 需要 Temperature = 0 |
| **原地** | 不依赖外部存储 | 不依赖外部检索 |

**探讨**：
- 自回归生成："单步 O(1)" + "稳定（T=0）" + "非原地（依赖历史）"
- Beam search："非 O(1)" + "不稳定" + "非原地"
- 是否存在"三元悖"？

## 层次三：Agent 设计练习

### E7. 策略选择 Skill

**任务**：设计一个 Skill：输入数据特征（规模、分布、键类型），自动在确定性排序、随机化排序和哈希之间选择策略。

**Skill 设计**：

```python
def choose_sort_strategy(n, key_type, key_range, stable_required, worst_case_required):
    """
    输入：
    - n: 数据规模
    - key_type: 'integer' | 'float' | 'string'
    - key_range: 键值范围（如果已知）
    - stable_required: 是否需要稳定性
    - worst_case_required: 是否需要最坏保证
    
    输出：
    - 推荐的排序策略
    - 原因说明
    """
    
    # 线性排序优先（如果条件满足）
    if key_type == 'integer' and key_range <= n:
        return '计数排序', '键值范围有限，线性时间可行'
    
    if key_type == 'string':
        return '基数排序', '字符串可按字符位排序'
    
    # 比较排序
    if worst_case_required:
        if stable_required:
            return '归并排序', '稳定 + 最坏保证'
        else:
            return '堆排序', '原地 + 最坏保证'
    
    if stable_required:
        return '归并排序', '稳定优先'
    
    # 默认：随机化快排
    return '随机化快排', '期望最优，不依赖输入分布'
```

**测试用例**：
1. n=100万，整数年龄（0-150），不需要稳定
2. n=1万，浮点相关性得分，需要稳定
3. n=10亿，字符串 URL，需要最坏保证

### E8. 字符串匹配选择 Skill

**任务**：设计一个 Skill：根据模式数量、文本规模、是否需要最坏保证，自动选择字符串匹配算法。

**Skill 设计**：

```python
def choose_string_matching(n, m, num_patterns, worst_case_required):
    """
    输入：
    - n: 文本规模
    - m: 模式长度
    - num_patterns: 模式数量
    - worst_case_required: 是否需要最坏保证
    
    输出：
    - 推荐的算法
    - 原因说明
    """
    
    if num_patterns > 1:
        return 'Aho-Corasick', '多模式匹配最优'
    
    if worst_case_required:
        return 'KMP', '单模式，最坏保证'
    
    # 单模式，概率性可接受
    if n > 10**6 and m < 100:
        return 'Rabin-Karp', '大规模文本，概率性快速匹配'
    
    return '朴素匹配', '规模小，实现简单'
```

### E9. 中位数流 Skill

**任务**：设计一个 Skill，实时计算数据流的中位数。

**核心挑战**：数据逐步到达，不能存储全部数据

**方案**：两个堆（最大堆 + 最小堆）

```
数据流中位数 Skill：

维护两个堆：
- max_heap：存储较小的一半（堆顶是较小部分的最大值）
- min_heap：存储较大的一半（堆顶是较大部分的最小值）

策略：
- 新元素：先放入 max_heap
- 平衡：如果两个堆大小差 > 1，移动堆顶
- 中位数：
  - 如果大小相等：两个堆顶的平均
  - 否则：较大堆的堆顶

复杂度：每次插入 O(log n)，查询中位数 O(1)
```

## 综合挑战

### E10. 排序系统设计

**任务**：设计一个完整的排序系统，支持多种数据类型和需求。

**需求**：
- 支持整数、浮点、字符串
- 支持稳定性要求
- 支持最坏保证
- 支持增量插入（数据逐步到达）
- 支持大规模数据（n > 10^8）

**设计要点**：
1. 策略选择器（类似 E7）
2. 增量排序（插入排序 + 定期重构）
3. 外部排序（数据超过内存时）
4. 并行排序（利用多核）
