# 3.11 从哈希到 LLM 采样：随机性是设计，不是缺陷

## 学习目标

学完这一节，你能：

- 理解哈希设计原则如何应用于 LLM 采样策略
- 分析 Temperature、Top-k、Top-p 的本质——排序 + 截断 + 随机选择
- 判断不同场景下应该选择什么采样参数
- 设计"概率换取效率"的 LLM 输出验证策略

---

## 核心洞察

哈希算法教给我们一个深刻的设计原则：

> 随机性不是要消除的缺陷，而是可以主动引入的设计元素。

哈希用随机性换取期望性能：
- 期望 O(1) 查询
- 最坏 Θ(n) 冲撞（但概率可控）
- Bloom Filter 允许假阳性（多候选），拒绝假阴性（漏掉）

LLM 的采样策略也遵循同样的原则：

```
┌─────────────────────────────────────────────────────────────────────┐
│  哈希与 LLM 采样的设计对照                                           │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  哈希设计原则                LLM 采样策略                            │
│  ─────────────              ─────────────                          │
│                                                                     │
│  随机哈希函数         →      Temperature 随机性                      │
│  期望 O(1) 查询       →      期望覆盖全局解空间                      │
│  Bloom Filter 假阳性  →      Top-k 多候选采样                        │
│  最坏 Θ(n) 冲撞       →      最坏局部最优陷阱                        │
│  可验证假阳性         →      可验证采样结果                          │
│                                                                     │
│  核心共性：用概率换取效率，用验证保证质量                            │
└─────────────────────────────────────────────────────────────────────┘
```

## LLM 采样的本质：排序 + 截断 + 随机选择

LLM 输出的采样过程本质上是：

```
┌─────────────────────────────────────────────────────────────────────┐
│  LLM 采样三阶段                                                      │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  Step 1: 排序                                                       │
│  ├── LLM 输出 log-probabilities                                    │
│  ├── 按概率排序：从高到低                                           │
│  └── 这是"确定性排序"阶段                                          │
│                                                                     │
│  Step 2: 截断                                                       │
│  ├── Top-k: 保留前 k 个候选                                         │
│  ├── Top-p: 保留累积概率 ≥ p 的候选                                 │
│  ├── Min-p: 保留 ≥ 最大概率 × p 的候选                              │
│  └── 这是"线性截断"阶段                                            │
│                                                                     │
│  Step 3: 随机选择                                                   │
│  ├── Temperature 重整：p_i = exp(l_i/T) / Σ exp(l_j/T)             │
│  ├── 按重整后概率随机采样                                           │
│  └── 这是"随机化"阶段                                              │
│                                                                     │
│  总结：采样 = 排序 + 截断 + 随机选择                                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 1. 排序阶段：确定性排序

LLM 对每个 token 输出 log-probability。按概率排序后：

```
log-probs 排序示例：
token:    "apple"  "banana"  "cherry"  "date"  "elder"
log-prob:  -0.5     -1.2      -2.3      -3.0    -5.0
排名：      1        2         3         4       5
```

这是确定性排序——对给定输入，排序结果固定。

### 2. 截断阶段：线性截断

**Top-k 截断**：

```
保留前 k 个 token：
k=3 → ["apple", "banana", "cherry"]

类比：哈希表只保留"前 n 个位置"
```

**Top-p 截断**：

```
累积概率 ≥ p 的 token：
p=0.9 → 累积概率 apple(0.6) + banana(0.3) + cherry(0.05) = 0.95 ≥ 0.9
保留：["apple", "banana", "cherry"]

类比：哈希表根据"负载因子"调整容量
```

**Min-p 截断**（新方法）：

```
保留 ≥ 最大概率 × p 的 token：
max_prob = 0.6 (apple), p = 0.1 → threshold = 0.06
保留：所有概率 ≥ 0.06 的 token

类比：哈希表的"最小桶大小"阈值
```

### 3. 随机选择阶段：Temperature 重整

**Temperature = 1**：原概率分布

```
P(token) = exp(log-prob) / Σ exp(log-probs)
```

**Temperature < 1**：更确定性（尖锐分布）

```
高温 → 低温：概率分布变尖锐
T=0.5 → "apple" 概率更高，其他更低
```

**Temperature > 1**：更随机（平坦分布）

```
低温 → 高温：概率分布变平坦
T=2.0 → 各 token 概率更接近
```

**Temperature = 0**：确定性采样（选最大概率）

```
T=0 → 总是选 "apple"
类比：确定性哈希（无随机性）
```

## 为什么 Temperature > 0 是设计，不是缺陷？

### 确定性采样的问题

Temperature = 0（确定性采样）看起来"最好"——每次都选最高概率的 token。

但问题是：

```
┌─────────────────────────────────────────────────────────────────────┐
│  确定性采样的陷阱                                                    │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  问题 1: 重复陷阱                                                    │
│  ├── 某些 token 概率总是最高                                         │
│  ├── 确定性采样总是选它们                                            │
│  ├── 输出陷入重复循环                                               │
│  └── 类比：哈希冲突导致单一桶负载过重                               │
│                                                                     │
│  问题 2: 局部最优陷阱                                                │
│  ├── 每步选最高概率 token                                           │
│  ├── 但整体序列可能不是最优                                          │
│  ├── 贪心策略不一定全局最优                                          │
│  └── 类比：确定性快排被攻击退化                                      │
│                                                                     │
│  问题 3: 缺乏多样性                                                  │
│  ├── 相同输入总是相同输出                                            │
│  ├── 无法生成多个候选方案                                            │
│  ├── 无法探索不同可能                                                │
│  └── 类比：计数排序无法处理浮点键                                    │
└─────────────────────────────────────────────────────────────────────┘
```

### Temperature > 0 的设计意义

引入 Temperature 随机性换取：

| 收益 | 类比哈希 |
|------|----------|
| **多样性**：避免重复循环 | 哈希避免单一桶过载 |
| **覆盖性**：探索不同可能 | 哈希覆盖整个键空间 |
| **鲁棒性**：不被输入分布控制 | 全域散列不被攻击 |

**核心洞察**：

```
确定性方法有其极限 → 必须引入随机性突破

排序：确定性 Ω(n lg n) → 随机化期望 O(n lg n)（消除输入依赖）
哈希：确定性冲突可攻击 → 全域散列 P[冲突] ≤ 1/m（消除攻击）
LLM：确定性重复陷阱 → Temperature > 0（消除重复）
```

## Top-k 与 Bloom Filter 的设计类比

### Top-k：多候选采样

Top-k 保留前 k 个 token，然后随机采样。

```
Top-k = 5 → 保留 ["apple", "banana", "cherry", "date", "elder"]
从这 5 个中采样
```

**类比 Bloom Filter**：

```
┌─────────────────────────────────────────────────────────────────────┐
│  Top-k 与 Bloom Filter 对照                                         │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  Bloom Filter:                                                      │
│  ├── 假阳性可接受：返回"可能存在"                                   │
│  ├── 假阴性不可接受：不漏掉真正存在                                 │
│  ├── 后续验证：确认是否真的存在                                     │
│                                                                     │
│  Top-k:                                                             │
│  ├── 假阳性可接受：包含多个候选                                     │
│  ├── 假阴性不可接受：不漏掉最佳                                     │
│  ├── 后续验证：评估哪个候选最好                                     │
│                                                                     │
│  设计共性：多候选 + 后验证                                          │
└─────────────────────────────────────────────────────────────────────┘
```

### Top-p：动态截断

Top-p 保留累积概率 ≥ p 的 token。token 数量动态变化。

```
某些位置：概率集中 → Top-p 保留少量 token
某些位置：概率分散 → Top-p 保留大量 token
```

**类比哈希表负载因子**：

```
哈希表：负载因子决定扩容
Top-p：累积概率决定截断点

都是动态调整策略
```

## 采样策略选择指南

当 agent 给你采样策略建议时，你要判断：

| 场景 | Temperature | Top-k/p | 原因 |
|------|-------------|---------|------|
| **精确答案（如数学）** | T ≈ 0 | Top-k=1 | 需要确定性最优 |
| **创意生成（如写作）** | T > 1 | Top-p=0.9 | 需要多样性 |
| **代码生成** | T ≈ 0.5 | Top-k=10 | 平衡精确与多样 |
| **对话生成** | T ≈ 0.7 | Top-p=0.95 | 自然多样性 |
| **多方案探索** | T > 0 | Top-k=多 | 需要多候选 |

**判断要点**：

```
1. 是否需要多样性？ → Temperature > 0
2. 是否需要精确最优？ → Temperature = 0 + 多轮验证
3. 是否需要覆盖全局？ → Top-k + 多候选验证
4. 假阳性是否可接受？ → 多候选，验证筛
```

---

## 验证策略：概率换取效率

哈希的验证策略：假阳性 → 确认验证

LLM 的验证策略：假阳性（采样输出）→ 确认验证

### 验证方法

**1. 人工验证**：

```
LLM 输出 → 人工审核 → 确认/拒绝
类比：哈希假阳性 → 逐字符验证
```

**2. 自动验证**：

```
LLM 输出 → 自动测试 → 确认/拒绝
类比：哈希假阳性 → 数据库查询验证
```

**3. 多候选验证**：

```
Top-k 输出 → 多候选 → 选最佳
类比：Bloom Filter → 多候选 → 验证筛
```

### 验证成本分析

| 方法 | 成本 | 适用场景 |
|------|------|----------|
| **单输出 + 人工验证** | 高（人工时间） | 创意内容 |
| **Top-k + 自动验证** | 中（运行测试） | 代码生成 |
| **Top-k + 模型验证** | 低（模型评估） | 大规模生产 |

## 练习

### 层次一（基础）

1. **采样三阶段**：给定 log-probs [-0.5, -1.2, -2.3, -3.0, -5.0]，执行 Top-k=3 + Temperature=1 的采样。

2. **Temperature 效果**：计算 T=0.5 和 T=2 时，上述 log-probs 的重整概率。

3. **假阳性率计算**：Top-k=10 时，"最佳 token 被包含的概率"是多少？

### 层次二（LLM 协同）

4. **采样策略对比**：让 LLM 对不同场景（数学、创意、代码）推荐采样参数，你判断是否合理。

5. **验证设计**：让 LLM 设计一个"Top-k + 自动验证"的代码生成流程，分析其哈希类比。

### 层次三（Agent 设计）

6. **采样策略选择 Skill**：设计一个 Skill，根据生成场景自动选择 Temperature、Top-k/p 参数。
