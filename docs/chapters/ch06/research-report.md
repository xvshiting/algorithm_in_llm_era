## 二、经典 DP 问题谱系

### 2.1 序列 DP

| 问题 | 状态定义 | 递推式 | 时间 | 空间 |
|------|----------|--------|------|------|
| **LIS（最长递增子序列）** | dp[i] = 以 i 结尾的 LIS 长度 | dp[i] = max(dp[j]+1) ∀j<i, a[j]<a[i] | O(n²) → O(n log n) | O(n) |
| **LCS（最长公共子序列）** | dp[i,j] = 前缀 i,j 的 LCS 长度 | 见 Z-031 编辑距离框架 | O(mn) | O(mn) |
| **编辑距离** | D[i,j] = 将 x[1..i] 变为 y[1..j] 的最小代价 | 三操作框架（插入/删除/替换） | O(mn) | O(mn) |

### 2.2 区间 DP

| 问题 | 状态定义 | 递推式 | 时间 | 空间 |
|------|----------|--------|------|------|
| **矩阵链乘** | m[i,j] = A_i..A_j 最少乘法次数 | m[i,j] = min_{i≤k<j}(m[i,k]+m[k+1,j]+p_{i-1}p_k p_j) | Θ(n³) → Θ(n²) Knuth优化 | Θ(n²) |
| **石子合并** | dp[i,j] = 合并区间 i..j 的最小代价 | 类矩阵链乘，代价函数不同 | Θ(n³) | Θ(n²) |
| **最优 BST** | e[i,j] = 区间 i..j 最小期望搜索代价 | e[i,j] = min_{i≤k≤j}(e[i,k-1]+e[k+1,j]+w(i,j)) | Θ(n³) | Θ(n²) |

### 2.3 树形 DP

| 问题 | 状态定义 | 递推式 | 时间 |
|------|----------|--------|------|
| **树的最大独立集** | dp[v][0/1] = v 选/不选时子树最大独立集 | dp[v][0] = Σ max(dp[u][0],dp[u][1]); dp[v][1] = Σ dp[u][0] | O(n) |
| **树的直径** | 两次 DFS 或一次 DP | first[v] + second[v] 维护最长/次长路径 | O(n) |
| **树上背包** | dp[v][k] = v 子树选 k 个物品的最大价值 | 合并子树 dp，类似区间 DP | O(nk²) → O(nk) |

### 2.4 状态压缩 DP

| 问题 | 状态定义 | 递推式 | 时间 | 适用规模 |
|------|----------|--------|------|----------|
| **TSP（旅行商）** | dp[S][i] = 已访问集合 S，当前在 i 的最短路径 | dp[S][i] = min_{j∈S}(dp[S-{i}][j] + dist(j,i)) | O(2^n · n²) | n ≤ 20 |
| **集合覆盖** | dp[S] = 覆盖集合 S 的最小子集数 | 类 TSP | O(2^n · k) | n ≤ 20 |
| **Hamilton 路径** | dp[S][i] = 从起点到 i，已访问 S | 类 TSP | O(2^n · n²) | n ≤ 20 |

## 三、教材参考对比

### 3.1 CLRS Ch15 动态规划（证明导向）

**四步法**：
1. 刻画最优解的结构（最优子结构）
2. 递归定义最优解的值
3. 自底向上计算
4. 构造最优解（辅助表回溯）

**CLRS DP 章节**：
- 钢条切割：两种递推揭示不同子问题结构
- 矩阵链乘：完整推导 + Knuth优化
- DP要素：最优子结构 + 子问题重叠 + 子问题图
- LCS：作为编辑距离特例
- 最优BST：概率分布决定形状

**CLRS 独有贡献**：
- **剪贴证明**形式化最优子结构验证
- **子问题图**连接 DP 到图论（反拓扑序）
- 运行时间 = O(|V| + |E|) 的 DAG 视角
- 最长简单路径**没有**最优子结构的反例

### 3.2 Skiena Ch10 动态规划（设计导向）

**三步法**：
1. 将答案表述为递推式或递归算法
2. 证明不同参数值的数量是多项式界的
3. 指定求值顺序使部分结果在需要时已就绪

**Skiena DP 章节**：
- Fibonacci 三阶递进（递归→记忆化→自底向上）
- 编辑距离统一框架（四桩函数）
- LIS → 子串匹配变体
- 子集和（伪多项式）
- CFG 解析（CKY 算法）

**Skiena 独有贡献**：
- **可插拔框架**：一个 DP 表 + 四桩函数解决一族问题
- **路径重构一等公民**："知道距离是 5 很好，但具体是哪 5 步？"
- War Story："条码文本压缩"（贪心 8-20% → DP O(n)）
- War Story："权力平衡"（三相负载平衡，状态空间 min/max 切换）

### 3.3 Kleinberg Ch6 动态规划（问题导向）

**Kleinberg/Tardos《Algorithm Design》Ch6**：
- 区间调度 → 加权区间调度 → DP 引入
- 背包问题 → 子集和 → 连续变化
- 序列比对 → 编辑距离 → LCS
- 图中的最短路径 → Bellman-Ford DP 视角
- PERT 图 → DAG 最长路径

**Kleinberg 独有贡献**：
- **问题演化序列**：贪心可解 → 加权版本 → DP 必需
- 区间调度变体的完整谱系
- 图问题的 DP 视角（Bellman-Ford = DP）
- 带权重 DAG 的最长路径（有最优子结构）

## 六、各节 War Story 候选

### 6.1 01-recursive-to-dp.md

**War Story：Fibonacci 的"指数陷阱"**

```
场景：面试现场，要求实现 Fibonacci
候选人 A：直接递归 → O(1.6^n)，n=40 要跑几秒
候选人 B：记忆化 → O(n)，秒算 n=100
候选人 C：自底向上 + 状态压缩 → O(n) 空间 O(1)

教训：递归看起来优雅，但子问题重叠使它爆炸
转折：缓存只在参数空间小且值被多次复用时有效
      - Fibonacci：有效（每个 n 被多次调用）
      - QuickSort：无效（每次参数都不同）
```

**核心叙事钩子**："同一个问题，三种实现，效率相差指数级。差别在哪？"

### 6.2 02-optimal-substructure.md

**War Story：最长路径的"剪贴失败"**

```
场景：求解图的最长简单路径
尝试：用 DP，状态 = 当前位置 + 已访问集合
问题：两个最长子路径可能共享顶点，拼接后不是简单路径
      例如：a-b-c-d 长度6，子路径 a→c 和 c→d 共享 c

剪贴证明失败：
  假设 a→c 的子路径不是最优 → 替换后
  新路径可能共享顶点 → 不是简单路径 → 无法比较

教训：最优子结构不是自动成立的，需要验证
      最长简单路径 NP 完全 → DP 不适用
      最短路有最优子结构 → DP 适用（Dijkstra/Bellman-Ford）
```

**核心叙事钩子**："为什么最短路可以用 DP，最长路径不行？剪贴证明揭示答案。"

### 6.3 03-edit-distance.md

**War Story：Skiena 的"四桩函数框架"**

```
场景：实现字符串比对系统
需求：支持多种比对模式（编辑距离、子串匹配、LCS）
朴素方案：每个模式单独实现 → 代码重复

Skiena 的方案：
  一个 DP 表 + 四个可插拔桩函数：
  - row_init()：初始化第一行
  - column_init()：初始化第一列
  - match(c1, c2)：匹配/替换代价
  - goal_cell()：终点位置
  
  改桩函数 → 改问题：
  - 标准编辑距离：全初始化为0，match代价可调
  - 子串匹配：column_init=0（自由起始位置）
  - LCS：禁止替换（match代价=0或∞）

教训：框架抽象比单独实现更强大
      "一个算法解决一族问题" → 可扩展性
```

**核心叙事钩子**："四个桩函数，一族问题。框架的威力在哪？"

### 6.4 04-classic-dp.md

**War Story 1：矩阵链乘的"括号艺术"**

```
场景：计算 A₁·A₂·A₃·A₄，各矩阵尺寸不同
  A₁: 30×35, A₂: 35×15, A₃: 15×5, A₄: 5×10

按序计算：
  (A₁A₂)A₃A₄：30×35×15 + 30×15×5 + 30×5×10 = 15750+2250+1500 = 19500
  实际最优需要完整枚举

DP 结果：
  ((A₁A₂)(A₃A₄))：30×35×15 + 15×5×10 + 30×15×10 = 15750+750+4500 = 21000
  等等，让我重算...

正确计算：
  A₁A₂: 30×35×15 = 15750, 结果 30×15
  A₃A₄: 15×5×10 = 750, 结果 15×10
  (A₁A₂)(A₃A₄): 30×15×10 = 4500
  总计：15750+750+4500 = 21000

按序 ((A₁(A₂A₃))A₄)：
  A₂A₃: 35×15×5 = 2625, 结果 35×5
  A₁(A₂A₃): 30×35×5 = 5250, 结果 30×5
  (A₁(A₂A₃))A₄: 30×5×10 = 1500
  总计：2625+5250+1500 = 9375

最优解：(A₁((A₂A₃)A₄)) = ?
  A₂A₃: 2625
  (A₂A₃)A₄: 35×5×10 = 1750, 结果 35×10
  A₁((A₂A₃)A₄): 30×35×10 = 10500
  总计：2625+1750+10500 = 14875

教训：括号位置影响乘法次数达 2x+
      DP 找最优括号化 Θ(n³)
      Knuth 优化可降至 Θ(n²)
```

**War Story 2：最优 BST 的"概率决定形状"**

```
场景：设计搜索引擎的词汇索引 BST
朴素方案：平衡树（AVL/红黑树）→ 高度最小

问题：搜索频率不均匀
  "algorithm" 搜索频率 30%
  "zebra" 搜索频率 0.1%

DP 方案：最优 BST
  状态：e[i,j] = 区间 i..j 的最小期望搜索代价
  目标：最小化 Σ freq[k] · (depth[k]+1)

结果：高频词靠近根，低频词在深处
      即使树高度更高，期望搜索代价更低

教训：最优解 ≠ 最"直观"的解
      约束（搜索频率）决定形状，而非高度平衡
```

### 6.5 05-dp-boundaries.md

**War Story：TSP 的"状态爆炸"**

```
场景：旅行商问题，n 个城市
尝试：用 DP，状态 = 已访问集合 S + 当前位置 i
      dp[S][i] = min_{j∈S}(dp[S-{i}][j] + dist(j,i))

状态数：2^n · n
      n=10 → 10240 状态，可行
      n=20 → ~1M 状态，勉强可行
      n=30 → ~1B 状态，不可行

教训：状态空间必须多项式有界
      TSP 的 O(2^n) 使 DP 退化为"指数暴力"
      比 n! 暴力好，但仍不可处理

对比：有序约束问题的状态空间多项式
      书本分页：O(kn²)，k=章节数，可行
```

### 6.6 06-dp-and-llm.md

**War Story：KV-Cache 的"记忆化重生"**

```
场景：LLM 推理时，用户问多个相关问题
问题：每次重新计算整个 context → 重复计算

KV-Cache 方案：
  存储已计算的 attention (K, V)
  共享 prefix → 复用计算 → 加速推理

类比 Fibonacci 记忆化：
  Fib(n) 递归 → 子问题重叠 → 记忆化加速
  LLM 推理 → token 依赖重叠 → KV-Cache 加速

关键条件：
  Fibonacci：参数 n 被多次调用 → 缓存有效
  QuickSort：每次参数不同 → 缓存无效
  KV-Cache：prefix 被多次查询 → 缓存有效
            完全不同的 prompt → 缓存无效

教训：记忆化的有效条件相同
      "子问题重叠" → "prefix reuse"
      理解 DP → 理解 KV-Cache 设计决策
```

### 7.2 复杂度演进表

| 优化技术 | 原复杂度 | 优化后 | 适用条件 |

| **记忆化** | 指数递归 | 多项式 | 子问题重叠 |
| **状态压缩** | O(n²) 空间 | O(n) | 只需前一状态 |
| **Knuth 优化** | Θ(n³) | Θ(n²) | 最优分割点单调 |
| **四边形不等式** | Θ(n³) | Θ(n²) | 满足四边形不等式 |
| **单调队列优化** | O(nk) | O(n) | 递推式特定形式 |

## 九、关键概念预告

### 9.1 三个"为什么"

1. **为什么 DP 记忆化有效？**
   - 条件：子问题重叠 + 参数空间多项式有界
   - 反例：QuickSort（每次参数不同）→ 记忆化无效

2. **为什么最长简单路径不能用 DP？**
   - 原因：没有最优子结构（两条子路径可能共享顶点）
   - 教训：最优子结构需要验证，不是自动成立

3. **为什么 KV-Cache 和 DP 记忆化同构？**
   - 条件相同：prefix 被多次查询 = 子问题重叠
   - 有效模式识别需要人的判断

### 9.2 三个"如何"

1. **如何验证最优子结构？**
   - 剪贴证明：假设子解非最优 → 替换 → 矛盾
   - 反例构造：找两个最优子解拼接后不合法

2. **如何设计 DP 状态？**
   - 子问题图视角：状态 = 子问题，边 = 依赖
   - 反拓扑序求值 → 自底向上实现

3. **如何将 DP 思维迁移到 LLM？**
   - 状态空间设计 → KV-Cache 管理
   - 子问题重叠识别 → prefix reuse 检测
   - 最优子结构验证 → LLM 输出评判

## 十三、教材写作风格参考

### 13.1 Kleinberg & Tardos 《Algorithm Design》DP 章节风格

**问题驱动结构（Ch6 动态规划）**：

Kleinberg & Tardos 以**加权区间调度**作为 DP 入门问题：
- 先讲无权区间调度（贪心可解）→ 学生理解问题
- 再引入权重 → 贪心失效 → 需要 DP
- **问题演化谱系**：同一问题从"贪心可解"到"需要 DP"，揭示 DP 的必要性

**写作特色**：

1. **问题开场**：不从"DP 定义"开始，从"贪心为什么失效"开始
2. **朴素→错误→优化**：
   - 朴素递归 → 指数爆炸
   - 加权版本 → 贪心失效
   - DP 记忆化 → 多项式解
3. **直觉先于形式**：
   - 先展示递归树（直觉）
   - 再给出递推式（形式）
   - 最后证明最优子结构（严谨）
4. **练习设计避免固定结构**：
   - 不分"基础/进阶/挑战"
   - 练习从问题变体自然生长：
     - 区间调度 → 加权区间调度 → 多机调度
     - 背包 → 子集和 → 连续变化

**DP 问题谱系（Kleinberg）**：

| 问题 | 引入方式 | 演化路径 |
|------|----------|----------|
| 加权区间调度 | 贪心失效 | 区间调度 → 加权 → 多机 |
| 背包问题 | 子集和变体 | 0-1背包 → 分数背包 → 子集和 |
| 序列比对 | 编辑距离 | LCS → 编辑距离 → 子串匹配 |
| 图最短路径 | Bellman-Ford | DAG最长 → 带环最短 |
| PERT 图 | DAG 任务 | DAG最长路径 → 关键路径 |

**关键启发**：
- **问题演化揭示算法必要性**：同一问题，贪心失效 → DP 必需
- **朴素方案展示错误**：让学生理解"为什么不能贪心"
- **直觉建立先用图**：递归树图解优于公式

```
War Story：最长路径的剪贴失败

问题：求图的最长简单路径
尝试：用 DP，状态 = 当前位置 + 已访问集合
困惑：为什么最短路可以用 DP，最长路径不行？
```

**展示错误**：
```
反例构造：
  图 a-b-c-d，边权 ab=2, bc=2, cd=2, ad=1
  a→d 最长路径 a-b-c-d 长度 6
  
  子路径 a→c 最长是 a-b-c 长度 4
  但 c→d 最长是 c-d 长度 2（不能走 a）
  
  拼接后：a-b-c-d ✓ 但不是最优拼接方式
  问题：两条子路径可能共享顶点
```

**直觉→形式化**：
```
直觉：
  最短路 = 子路最短 + 子路最短 ✓
  最长路 = 子路最长 + 子路最长 ✗

剪贴证明（形式化）：
  假设最优解包含非最优子解
  剪下替换 → 得到更好的解 → 矛盾
  
  最短路成立：替换后仍是简单路径
  最长路不成立：替换后可能共享顶点
```

### 15.3 03-edit-distance.md 写作风格

**问题开场**：
```
War Story：面试现场的编辑距离优化

问题：实现字符串比对系统，支持多种模式
传统方案：每个模式单独实现 → 代码重复
```

**朴素→错误→优化**：
```
1. 朴素方案：为 LCS、编辑距离、子串匹配各写一个 DP
2. 问题暴露：代码重复，维护困难
3. 框架方案：一个 DP 表 + 四桩函数
```

**直觉先于形式**：
```
直觉：
  "都是字符串比对，核心逻辑相同"
  "差异只是代价函数和边界条件"

框架（形式化）：
  四个桩函数：
    row_init() → 第一行初始化
    column_init() → 第一列初始化
    match(c1, c2) → 匹配/替换代价
    goal_cell() → 终点位置
  
  改桩函数 → 改问题
```

### 16.2 War Story 叙事结构强化

**每个 War Story 必须包含**：

1. **真实场景**（工程或面试）
2. **朴素方案**（看起来合理）
3. **问题暴露**（数据对比震撼）
4. **DP 改进**（正确方案）
5. **教训总结**（抽象洞察）

**数据对比要求**：
- 必须有具体数字（如"1ms vs 10分钟"）
- 必须有步骤对比（如"1,904,832步 vs 48步"）

**不使用固定三层（基础/LLM协同/Agent）**
