学生需要理解的不是 DP 的定义，而是：
- 为什么同一个递归思想，在 Fibonacci 上失效，在归并排序上有效？

- 为什么"最优子结构"不是理所当然的？

```

def merge_sort(arr):
```

### 01-recurrence.md - 子问题与重叠：为什么分治失效

**问题情境**：Fibonacci 的"分治陷阱"

小明学完分治，用递归解 Fibonacci：
```
第 30 项：5 秒
第 40 项：55 秒
第 50 项：3 小时还没出结果...
```

小明困惑："分治思想是对的——分解成两个子问题，合并结果。为什么这么慢？"

**直观思路**：画出递归树

小红让小明画出 `fib(5)` 的调用树：
```
          fib(5)
         /      \
     fib(4)    fib(3)
    /    \     /    \
 fib(3) fib(2) fib(2) fib(1)
  /  \
fib(2) fib(1)
```

小红指出："`fib(3)` 被计算了 **两次**。`fib(2)` 被计算了 **三次**。"

**关键洞察**：叶子节点数 ≈ φ^n（φ = 1.618...），这是指数级！

**对比归并排序**：
```
        merge_sort(n)
         /        \
 merge_sort(n/2)  merge_sort(n/2)
```

- 左半和右半**不重叠**
- 每个子问题只计算一次
- 时间 = O(n log n)，不是指数

**困惑解答**：分治有效的前提是**子问题独立**。Fibonacci 的子问题**重叠**，所以分治失效。

**规范定义**：

**子问题图（Subproblem Graph）**：
- 顶点 = 所有子问题
- 边 = 子问题之间的依赖关系
- Fibonacci 的子问题图是 DAG（有向无环图）

**重叠子问题**：
- 如果子问题图中有多个路径到达同一个顶点
- 则该子问题会被多次计算

**时间复杂度公式**：
```
时间 ≈ 子问题数 × 每个子问题的计算代价
```

- Fibonacci 朴素递归：子问题数 = φ^n，计算代价 = O(1)，总时间 = O(φ^n)
- 归并排序：子问题数 = n，计算代价 = O(n)（合并），总时间 = O(n log n)

**算法实现**：

**方案 A：记忆化递归**（自顶向下）
```python
def fib_memo(n, memo={}):
    if n in memo: return memo[n]
    if n <= 1: return n
    memo[n] = fib_memo(n-1, memo) + fib_memo(n-2, memo)
    return memo[n]
```

**方案 B：迭代填表**（自底向上）
```python
def fib_dp(n):
    dp = [0] * (n + 1)
    dp[0], dp[1] = 0, 1
    for i in range(2, n + 1):
        dp[i] = dp[i-1] + dp[i-2]
    return dp[n]
```

**方案 C：空间优化**
```python
def fib_optimized(n):
    if n <= 1: return n
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b
```

**正确性分析**：

记忆化不改变计算结果，只改变计算顺序：
- 朴素递归：先计算 fib(5)，发现需要 fib(4) 和 fib(3)，再往下...
- 记忆化：第一次计算 fib(3) 时存储结果，第二次直接返回

**证明**：每次调用 `fib_memo(n, memo)` 返回的值与 `fib(n)` 相同，因为：
1. 递推公式相同：`f(n) = f(n-1) + f(n-2)`
2. 边界条件相同：`f(0) = 0, f(1) = 1`
3. memo 只是缓存，不影响逻辑

**复杂度分析**：

| 方法 | 时间 | 空间 |
|-----|-----|-----|
| 朴素递归 | O(φ^n) ≈ O(1.618^n) | O(n) 栈深度 |
| 记忆化递归 | O(n) | O(n) memo + O(n) 栈 |
| 迭代填表 | O(n) | O(n) |
| 空间优化 | O(n) | O(1) |

**为什么空间能优化到 O(1)？**
- Fibonacci 只依赖前两项
- 不需要保存整个表，只需要保存最近两个值

**变式练习**：

**基本理解**：
1. 画出 `fib(7)` 的递归树，标记哪些节点被重复计算
2. 计算朴素递归需要多少次 `fib(1)` 调用？

**方法应用**：
3. 实现记忆化的 Fibonacci，对比朴素递归的运行时间
4. 为什么 quicksort 不需要记忆化？（画出子问题图）

**错误诊断**：
5. 如果有人提议给归并排序加记忆化，你会怎么反驳？
6. 有人用记忆化解 Fibonacci，但每次调用都创建新的 memo 字典，效率还是很低。问题在哪里？

**方案比较**：
7. 记忆化递归 vs 迭代填表：哪种更适合？什么情况下选择哪种？
8. 如果 Fibonacci 的递推改成 `f(n) = f(n-1) + f(n-2) + f(n-3)`，空间优化还有效吗？

**开放设计**：
9. 设计一个"子问题是否重叠"的判断标准：给定问题描述，判断是否需要记忆化
10. 如果子问题图不是 DAG（有环），会发生什么？能举例吗？

**为什么/能否改进**：
11. 为什么 Fibonacci 的递归树是指数级，但记忆化后是线性？
12. 能否设计一个算法，自动检测子问题重叠？

### 02-optimal.md - 最优子结构：为什么拼接会出错

**问题情境**：最长简单路径的"拼接陷阱"

小红用 DP 解最长简单路径问题：

```
问题：在图中找 A→D 的最长简单路径（不能重复顶点）

小红的设计：
- 分解成 A→C 的最长路径 + C→D 的最长路径
- 递推：longest(A,D) = longest(A,C) + longest(C,D)

结果：答案错误！
```

小红困惑："我用了 DP 的分解思想，为什么拼接起来就不对了？"

**直观思路**：构造反例

小明画了一个图：
```
A → B → C → D
    ↓
    E → D
```

- A→D 的最长简单路径是 **A→B→C→D**（长度 3）
- A→C 的最长简单路径是 **A→B→C**（长度 2）
- C→D 的最长简单路径是 **C→D**（长度 1）
- 拼接：A→B→C + C→D = A→B→C→D ✓... 看起来对？

小明再加一条边：
```
A → B → C → D
    ↓
    E → C → D
```

- A→C 的最长简单路径可能是 **A→E→C**（长度 2）
- C→D 的最长简单路径是 **C→D**（长度 1）
- 拼接：A→E→C + C→D = A→E→C→D... 
- 但 **A→B→C→D** 更长（长度 3）！

**关键洞察**：最长简单路径不能重复顶点。如果 A→C 用了某个顶点，C→D 就不能再用。

**对比最短路径**：
- 最短路径允许重复顶点（如果是简单路径，不重复也没问题）
- A→C 的最短路径 + C→D 的最短路径 = A→D 的最短路径 ✓

**困惑解答**：DP 分解有效的前提是**最优子结构**——子问题的最优解拼接后仍是原问题的最优解。最长简单路径**没有最优子结构**。

**规范定义**：

**最优子结构**：
如果问题的最优解包含子问题的最优解，则该问题具有最优子结构。

形式化：
```
设 OPT(X) 为问题 X 的最优解
如果 X 可分解为子问题 Y 和 Z
且 OPT(X) = OPT(Y) + OPT(Z)
则问题 X 具有最优子结构
```

**剪贴证明法**（CLRS）：

假设子问题的解不是最优的：
1. 假设 OPT(X) 包含的 Y 的解不是 OPT(Y)，而是某个 Y'
2. 剪下 Y'，贴上 OPT(Y)
3. 新解 OPT(Y) + OPT(Z) 应该比原解更好
4. 这与 OPT(X) 是最优解矛盾

**证明最短路径有最优子结构**：

设 A→D 的最短路径经过中间点 C：
- 如果 A→C 不是最短路径，存在更短的 A→C 路径
- 替换后，A→D 的路径会更短
- 这与"原路径是最短路径"矛盾 ✓

**证明最长简单路径没有最优子结构**：

设 A→D 的最长简单路径经过中间点 C：
- A→C 的最长路径可能用了某个顶点 V
- C→D 的最长路径也可能用了顶点 V
- 拼接后，V 重复，不是简单路径
- 剪贴法失效 ✗

**算法实现**：

最短路径的 DP 递推（Bellman-Ford 思想）：
```python
def shortest_path_dp(graph, start, end, k):
    """最多经过 k 条边的最短路径"""
    if k == 0:
        return 0 if start == end else infinity
    if start == end:
        return 0
    
    min_dist = shortest_path_dp(graph, start, end, k-1)
    for neighbor in graph[start]:
        dist = graph[start][neighbor] + shortest_path_dp(graph, neighbor, end, k-1)
        min_dist = min(min_dist, dist)
    return min_dist
```

最长简单路径：不能用 DP，需要其他方法（如搜索）。

**正确性分析**：

剪贴证明的本质是：**最优解的局部也必须是最优的**。

- 如果局部不是最优，可以替换它，让全局更好
- 这意味着"局部最优"是"全局最优"的必要条件

**最长简单路径为什么失败**：
- "局部最优"拼接后，可能违反全局约束（不能重复顶点）
- 局部和全局之间有"隐式依赖"，剪贴证明没考虑到

**复杂度分析**：

| 问题 | 最优子结构 | DP 时间 | 其他方法时间 |
|-----|----------|---------|------------|
| 最短路径 | ✓ | O(n²) Bellman-Ford | O(n²) Dijkstra |
| 最长简单路径 | ✗ | 不适用 | O(n!) 搜索 |

**变式练习**：

**基本理解**：
1. 用剪贴证明论证钢条切割的最优子结构
2. 最短路径为什么有最优子结构，而最长简单路径没有？

**方法应用**：
3. 判断 0-1 背包是否有最优子结构，用剪贴证明
4. 判断分数背包是否有最优子结构，它为什么能用贪心？

**错误诊断**：
5. 构造一个最长简单路径的反例，说明拼接失败
6. 有人设计 DP 解"最长路径"（允许重复顶点），结果算法正确但没有最优解（可能无限长）。问题在哪里？

**方案比较**：
7. 最短路径：DP（Bellman-Ford）vs 贪心（Dijkstra）vs BFS。各有什么适用条件？
8. 0-1 背包 vs 分数背包：为什么一个需要 DP，一个可以用贪心？

**开放设计**：
9. 设计一个"最优子结构检测器"：给定问题描述，判断是否有最优子结构
10. 有没有"部分最优子结构"的问题？（某些分解方式有效，某些无效）

**为什么/能否改进**：
11. 为什么最长简单路径的反例中，顶点重复导致拼接失败？
12. 如果放宽"简单路径"的定义（允许重复 k 个顶点），能否用 DP？

### 03-state-design.md - 状态设计：维度爆炸的困境

**问题情境**：比赛时的"状态陷阱"

小明在算法竞赛看到题目：
```
问题：给定 n 个物品，每个有重量 w[i] 和价值 v[i]，背包容量 W
求：能装的最大价值（每个物品最多用一次）
```

小明直觉知道是 DP，但设计了 5 维状态数组：
```python
dp[i][j][k][l][m] = ...  # 空间直接爆了！
```

评委提醒："状态设计要考虑参数的有效组合，不是简单堆砌变量。"

**直观思路**：从问题定义出发

小红分析问题：
- 需要跟踪什么信息？
  - 已经考虑了哪些物品？→ 需要一个索引 i
  - 当前背包用了多少容量？→ 需要一个容量 j
- 其他信息重要吗？
  - 当前价值？→ 不需要，dp[i][j] 的值就是价值
  - 具体装了哪些物品？→ 不需要，只要价值最大

**状态精简**：`dp[i][j]` = 前 i 个物品，容量 j 时的最大价值

**状态转移**：
- 不装第 i 个物品：`dp[i][j] = dp[i-1][j]`
- 装第 i 个物品：`dp[i][j] = dp[i-1][j-w[i]] + v[i]`（如果 j >= w[i]）
- 取最大：`dp[i][j] = max(...)`

**关键洞察**：状态维度 = 真正需要跟踪的参数数量。不是所有参数都需要。

**规范定义**：

**状态空间**：
- 状态 = 描述子问题的一组参数
- 状态空间 = 所有可能的状态组合

**状态转移方程**：
- 描述从一个状态如何转移到另一个状态
- 本质是子问题的递推关系

**状态设计原则**：
1. **完整性**：状态必须唯一确定子问题
2. **最小性**：状态参数尽量少（减少空间）
3. **可达性**：所有状态都能从初始状态转移到达

**复杂度公式**：
```
时间 = 状态数 × 每个状态的转移代价
空间 = 状态数
```

**算法实现**：

**背包问题**：
```python
def knapsack_dp(weights, values, capacity):
    n = len(weights)
    dp = [[0] * (capacity + 1) for _ in range(n + 1)]
    
    for i in range(1, n + 1):
        for j in range(capacity + 1):
            # 不装第 i 个物品
            dp[i][j] = dp[i-1][j]
            # 装第 i 个物品（如果容量够）
            if j >= weights[i-1]:
                dp[i][j] = max(dp[i][j], 
                               dp[i-1][j-weights[i-1]] + values[i-1])
    
    return dp[n][capacity]
```

**LCS（最长公共子序列）**：
```python
def lcs_dp(s1, s2):
    m, n = len(s1), len(s2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s1[i-1] == s2[j-1]:
                dp[i][j] = dp[i-1][j-1] + 1
            else:
                dp[i][j] = max(dp[i-1][j], dp[i][j-1])
    
    return dp[m][n]
```

**正确性分析**：

状态设计正确性的验证：
1. **覆盖性**：是否所有情况都被状态覆盖？
2. **唯一性**：是否每个子问题只对应一个状态？
3. **转移正确性**：是否所有可能的转移都被考虑？

**背包问题的验证**：
- dp[i][j] 表示"前 i 个物品，容量 j"——唯一确定
- 转移考虑了"装"和"不装"两种情况——覆盖完整
- 取最大——保证最优

**复杂度分析**：

| 问题 | 状态维度 | 状态数 | 转移代价 | 总时间 | 空间 |
|-----|---------|-------|---------|-------|-----|
| Fibonacci | 1D | n | O(1) | O(n) | O(n) → O(1) |
| 背包 | 2D | n×W | O(1) | O(nW) | O(nW) → O(W) |
| LCS | 2D | m×n | O(1) | O(mn) | O(mn) → O(min(m,n)) |
| 矩阵链乘 | 2D+区间 | n² | O(n) | O(n³) | O(n²) |
| TSP | n×2^n | n·2^n | O(n) | O(n²·2^n) | O(n·2^n) |

**空间优化技巧**：
- 只依赖前一行的状态 → 可以用滚动数组
- 只依赖前 k 个状态 → 可以用循环队列

**变式练习**：

**基本理解**：
1. 背包问题为什么只需要 `dp[i][j]` 两个维度？
2. LCS 的状态 `dp[i][j]` 表示什么？

**方法应用**：
3. 设计"硬币兑换"问题的状态：给定金额和硬币面值，求最少硬币数
4. 设计"编辑距离"问题的状态：将字符串 A 变成 B 的最小代价

**错误诊断**：
5. 有人设计 `dp[i][j][v]` 表示"前 i 个物品，容量 j，价值 v"，维度爆炸。问题在哪里？
6. 有人设计状态时遗漏了某个转移，结果答案偏小。如何诊断？

### 04-classic-models.md - 经典模型：统一框架的力量

**问题情境**：面试时的"框架陷阱"

小明面试被问到编辑距离。他用递归实现，面试官说"优化空间"。小明改成滚动数组。面试官又问"如果只要答案，不要路径重构呢？"

小明困惑：每次都要重新设计，有没有通用框架？

**直观思路**：Skiena 的统一框架

小红指出：Skiena 提出了一个**统一框架**——编辑距离的四种操作对应四个可插拔桩函数：

```python
def edit_distance_dp(s1, s2, 
                     match_cost,    # 匹配的代价
                     insert_cost,   # 插入的代价
                     delete_cost,   # 删除的代价
                     replace_cost): # 替换的代价
    
    m, n = len(s1), len(s2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    
    # 初始化边界
    for i in range(m + 1):
        dp[i][0] = i * delete_cost  # 删除所有 s1 字符
    for j in range(n + 1):
        dp[0][j] = j * insert_cost  # 插入所有 s2 字符
    
    # 状态转移
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s1[i-1] == s2[j-1]:
                dp[i][j] = dp[i-1][j-1] + match_cost
            else:
                dp[i][j] = min(
                    dp[i-1][j] + delete_cost,      # 删除 s1[i-1]
                    dp[i][j-1] + insert_cost,      # 插入 s2[j-1]
                    dp[i-1][j-1] + replace_cost    # 替换
                )
    
    return dp[m][n]
```

**关键洞察**：修改桩函数，可以解决一族问题：
- `match_cost = 0, insert_cost = delete_cost = replace_cost = 1` → 标准编辑距离
- `match_cost = 1, insert_cost = delete_cost = replace_cost = -∞` → LCS
- `match_cost = 0, insert_cost = delete_cost = ∞, replace_cost = 1` → Hamming 距离

**规范定义**：

**钢条切割**：
- 状态：`r[n]` = 长度 n 的钢条最大收益
- 转移：`r[n] = max(p[i] + r[n-i])` for all i ≤ n

**矩阵链乘**：
- 状态：`m[i,j]` = 矩阵 i 到 j 的最小乘法次数
- 转移：`m[i,j] = min(m[i,k] + m[k+1,j] + p[i-1]·p[k]·p[j])` for all k
- Knuth 优化：分割点 k 单调递增 → O(n²)

**最优 BST**：
- 状态：`e[i,j]` = 键 i 到 j 的期望搜索代价
- 转移：`e[i,j] = min(e[i,k-1] + e[k+1,j] + w(i,j))` for all k
- w(i,j) = 概率之和

**算法实现**：

**钢条切割（两种实现）**：

自顶向下：
```python
def cut_rod_td(n, prices, memo={}):
    if n in memo: return memo[n]
    if n == 0: return 0
    max_val = -infinity
    for i in range(1, n + 1):
        max_val = max(max_val, prices[i] + cut_rod_td(n - i, prices, memo))
    memo[n] = max_val
    return max_val
```

自底向上：
```python
def cut_rod_bu(n, prices):
    dp = [0] * (n + 1)
    for j in range(1, n + 1):
        max_val = -infinity
        for i in range(1, j + 1):
            max_val = max(max_val, prices[i] + dp[j - i])
        dp[j] = max_val
    return dp[n]
```

**正确性分析**：

钢条切割的最优子结构：
- 假设长度 n 的最优切割是：第一刀切 i，剩余 n-i
- 如果剩余 n-i 的切割不是最优，可以替换成最优切割
- 总收益会更高，矛盾 ✓

**复杂度分析**：

| 问题 | 时间 | 空间 | 优化后时间 |
|-----|-----|-----|----------|
| 钢条切割 | O(n²) | O(n) | — |
| 矩阵链乘 | O(n³) | O(n²) | O(n²) Knuth |
| 编辑距离 | O(mn) | O(mn) | O(min(m,n)) 滚动 |
| 最优 BST | O(n³) | O(n²) | — |

### 05-llm-dp.md - LLM 与 DP 的同构性

**问题情境**：推理的"重复计算困境"

研究员发现：GPT-4 在复杂推理时，经常重复计算相同步骤：
```
问题：(A + B) × C + (A + B) × D

推理过程：
1. 计算 A + B = X
2. 计算 X × C
3. 计算 A + B = X（重复了！）
4. 计算 X × D
```

研究员加了 KV-cache，推理速度提升 10 倍。他惊讶："这不就是 DP 的记忆化吗？"

**直观思路**：推理步骤的子问题图

小红分析：
- LLM 推理是逐步生成
- 每一步依赖前面生成的 token
- 如果两个推理路径的"前缀"相同，后面的生成可以缓存

**类比**：
- Fibonacci：`fib(3)` 被多次计算 → 记忆化
- LLM 推理：相同前缀的生成 → KV-cache

**关键洞察**：KV-cache 本质是记忆化。但"子问题"变成了"公共前缀"。

**规范定义**：

**KV-cache 的机制**：
- 存储：每个位置的 Key 和 Value 向量
- 复用：相同前缀的位置不需要重新计算
- 本质：空间换时间

**DP 与 KV-cache 的对应**：

| DP 概念 | LLM 对应 |
|--------|---------|
| 子问题 | 公共前缀 |
| 记忆化 | KV-cache |
| 子问题重叠 | 推理步骤复用 |
| 状态设计 | Prompt 设计（影响缓存效率） |

**CoT 的子问题分析**：

```
问题：计算 (1+2+3) × 4

CoT 推理：
Step 1: 1 + 2 = 3
Step 2: 3 + 3 = 6
Step 3: 6 × 4 = 24

如果另一个问题是 (1+2+3) × 5：
Step 1-2 完全相同 → 可以复用 KV-cache
Step 3: 6 × 5 = 30（不同）
```

**ToT 的状态空间爆炸**：

Tree of Thoughts（思维树）：
- 每步生成 k 个候选
- 探索深度 d
- 状态数 ≈ k^d

类比：
- DP 的状态空间：可能是指数（如 TSP）
- ToT 的状态空间：也是指数
- 优化方法：剪枝（类似 DP 的剪枝条件）

**正确性分析**：

KV-cache 正确性的前提：
1. 相同前缀 → 相同的 Key/Value
2. 注意力计算依赖所有位置 → 缓存必须完整

**复杂度分析**：

| 方法 | 时间 | 空间 |
|-----|-----|-----|
| 无缓存推理 | O(n²)（每个位置要计算注意力） | O(1) |
| 有缓存推理 | O(n)（只计算新位置） | O(n) |

**变式练习**：

**基本理解**：
1. KV-cache 为什么能加速 LLM 推理？
2. KV-cache 与 Fibonacci 记忆化的类比是什么？

**方法应用**：
3. 分析 CoT 推理的"子问题重叠"：哪些步骤可以复用？
4. 设计一个推理缓存策略：哪些中间结果应该缓存？

**错误诊断**：
5. 如果 KV-cache 的位置顺序变了（如 reordered），会发生什么？
6. 如果前缀相同但 context 不同（如不同任务），缓存还有效吗？

**方案比较**：
7. KV-cache vs 无缓存：空间换时间的权衡是什么？
8. ToT vs CoT：状态空间爆炸的风险是什么？

**开放设计**：

**为什么/能否改进**：
11. 为什么 LLM 推理可以用 DP 思想优化，但推理本身不是 DP？
12. 能否用 DP 的状态空间分析预测 ToT 的计算成本？
