# 第9章 从单机到分布式：规模改变算法

## 本章核心问题

前面八章默认的计算环境大多是单机：数据在同一块内存里，操作按一个清晰的顺序发生，错误通常来自算法设计本身。

真实系统会把这个前提打碎：

```text
数据不在一台机器上。
消息会延迟、丢失或乱序。
节点可能失败。
局部最优的通信策略可能拖垮整体吞吐。
```

第9章讨论规模改变之后，算法分析怎样从“计算步骤”扩展到“通信、同步、容错和一致性”。

---

## 9.1 为什么分布式不是“把代码多跑几份”

把一个程序复制到多台机器上，并不自动得到一个分布式算法。真正困难的是：

- 每个节点只知道局部信息。
- 节点之间通信有成本。
- 没有一个永远可靠、永远最新的全局状态。
- 同一个事件在不同机器上看到的顺序可能不同。

生活中可以把它想成多人协作整理一座大型图书馆。每个人只负责一个区域，如果每放一本书都跑去总台确认，系统会很慢；如果完全不沟通，书目又会不一致。分布式算法要做的，就是在沟通成本和一致性之间找到可证明的规则。

---

## 9.2 通信复杂度：新的成本模型

单机算法常问：

```text
做了多少次比较？
访问了多少个元素？
用了多少额外内存？
```

分布式环境还要问：

```text
发送了多少条消息？
每条消息多大？
需要多少轮通信？
是否存在中心节点瓶颈？
节点失败后是否能恢复？
```

一个 `O(n)` 的本地扫描，如果需要把所有数据都传到中心机器，可能被网络带宽卡住。相反，一个本地多做一点计算、少发很多消息的算法，在真实系统中可能更快。

---

## 9.3 MapReduce：用受限模型换取规模

MapReduce 把大规模处理拆成两个动作：

```text
map：把输入记录变成中间键值对。
reduce：把相同 key 的中间值聚合起来。
```

它牺牲了一部分表达能力，换来了清晰的数据流和容错边界。词频统计是最经典的例子：

```python
def mapper(line):
    for word in line.split():
        yield word, 1

def reducer(word, counts):
    yield word, sum(counts)
```

真正的算法问题不在代码行数，而在键的选择、数据倾斜、shuffle 成本和 reduce 端是否会成为热点。

---

## 9.4 一致性：什么时候必须看到同一个世界

很多学生第一次接触分布式系统时，会把一致性理解成“所有机器数据一样”。更准确地说，一致性是系统对读写顺序作出的承诺。

例如课程报名系统：

- 如果两个学生同时抢最后一个名额，是否允许都看到“还有 1 个名额”？
- 如果老师刚发布公告，学生刷新页面是否必须立刻看到？
- 如果统计面板延迟 10 秒更新，是否可以接受？

不同问题需要不同强度的一致性。算法设计要先问清业务语义，再选择锁、事务、共识、幂等重试或最终一致。

---

## 9.5 图算法在分布式中的新形态

图算法进入分布式环境后，难点从“怎样遍历边”变成：

```text
边分布在哪些机器上？
一个顶点的邻居是否跨越多个分片？
每一轮传播需要多少网络通信？
什么时候可以停止？
```

PageRank、连通分量、最短路、社区发现都可以看作消息传播过程。你在第4章学过的图建模仍然重要，但现在还要审查数据分区和通信轮数。

---

## 9.6 Agent 时代的分布式审查

Coding agent 容易给出“功能正确但系统不可用”的方案。例如：

```text
把所有日志拉到一台机器排序，然后统计。
```

这个方案在小样例里很好，在真实规模下可能直接失败。审查时可以要求 agent 补全：

```text
请给出单机方案和分布式方案。
分别分析计算成本、网络传输、存储成本、数据倾斜、失败恢复和一致性要求。
如果使用 MapReduce / 分片 / 缓存，请说明 key 的设计和热点风险。
```
