## 三、分布式算法核心内容分析

### 3.1 通信复杂度：新的成本模型

**核心叙事**：
> 单机算法问"做了多少次计算"，分布式算法还要问"发送了多少条消息"。

**教学引入建议**：
```
┌─────────────────────────────────────────────────────────────────────┐
│  成本模型的演变                                                       │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  单机算法：                                                          │
│  ├── 计算复杂度：O(n) 次操作                                        │
│  ├── 内存复杂度：O(n) 额外空间                                      │
│  └── 前提：数据在同一块内存                                         │
│                                                                     │
│  分布式算法：                                                        │
│  ├── 计算复杂度：每个节点本地计算                                   │
│  ├── 通信复杂度：发送多少消息？每条多大？                           │
│  ├── 同步成本：需要多少轮通信？                                     │
│  └── 前提：数据分布在多台机器                                       │
│                                                                     │
│  关键洞察：                                                          │
│  1. O(n) 本地扫描 vs 传输所有数据到中心 → 网络带宽瓶颈              │
│  2. 本地多计算 + 少发消息 → 可能更快                                │
│  3. 没有永远可靠、永远最新的全局状态                                │
└─────────────────────────────────────────────────────────────────────┘
```

**经典案例对比**：

| 问题 | 单机方案 | 分布式方案 | 通信成本考量 |
|------|---------|-----------|------------|
| **词频统计** | 单机扫描O(n) | MapReduce | Shuffle成本 |
| **最短路径** | Dijkstra O(V²) | 分布式Bellman-Ford | 每轮传播消息 |
| **连通分量** | DFS/BFS O(V+E) | Label Propagation | 标签传播轮数 |
| **PageRank** | 矩阵迭代O(V²) | 分布式迭代 | 每轮广播排名 |

### 3.2 MapReduce：用受限模型换取规模

**核心思想**：
> MapReduce牺牲了一部分表达能力，换来了清晰的数据流和容错边界。

**直觉引入**：
```
问题：统计1亿篇文章的词频

方案1（单机思路）：把所有文章传到一台机器，扫描统计
  ├── 传输1亿篇文章 → 网络瓶颈
  ├── 单机内存不够 → 频繁swap
  └── 一台机器故障 → 全部重来

方案2（MapReduce）：每台机器处理本地文章，聚合结果
  ├── Map：本地统计，只发(word, count)对
  ├── Shuffle：按word分发到对应节点
  ├── Reduce：每个word的count加总
  └── 容错：某节点失败 → 只重跑该节点
```

**算法问题不在代码，在于设计**：
1. **键的选择**：如何避免热点？
2. **数据倾斜**：某键过多 → Reduce端瓶颈
3. **Shuffle成本**：中间数据量决定性能
4. **Combiner优化**：Map端预聚合减少传输

### 3.3 数据分片、负载均衡与热点

**核心问题**：如何将数据分配到多台机器？

**分片策略对比**：

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **哈希分片** | 均匀分布 | 范围查询困难 | KV存储 |
| **范围分片** | 范围查询高效 | 可能热点 | 时间序列 |
| **一致性哈希** | 动态扩缩容 | 复杂度高 | 缓存系统 |
| **图分片** | 保局部性 | 边跨节点多 | 社交网络 |

**热点问题案例**：

```
案例：社交网络"名人"节点
  ├── 某用户有百万粉丝 → 该节点负载极高
  ├── 哈希分片 → 该节点所在的机器被压垮
  └── 解决方案：
      1. 热点识别：监控节点访问频率
      2. 热点分裂：将该节点复制到多台机器
      3. 读写分离：写主节点，读复制节点
```

### 3.4 一致性、幂等性与容错

**核心叙事**：
> 一致性不是"所有机器数据一样"，而是"系统对读写顺序作出的承诺"。

**一致性级别阶梯**：

| 级别 | 定义 | 性能代价 | 适用场景 |
|------|------|---------|---------|
| **线性一致性** | 所有操作看起来瞬时发生 | 最高 | 金融交易 |
| **顺序一致性** | 所有进程看到相同顺序 | 高 | 配置管理 |
| **因果一致性** | 因果相关操作顺序一致 | 中 | 社交评论 |
| **最终一致性** | 最终收敛到相同状态 | 低 | 缓存、日志 |

**经典案例：课程报名系统**：

```
场景：最后1个名额，2个学生同时抢

强一致方案：
  ├── 先到先得，用锁/事务保证
  └── 性能代价：每次操作都要同步

最终一致方案：
  ├── 都允许报名，后台检测超额
  ├── 超额后随机取消一个（公平？）
  └── 性能优势：本地操作，异步同步

设计决策：
  ├── 需要强一致吗？看业务语义
  ├── 老师刚发布公告，学生刷新必须立刻看到吗？
  ├── 统计面板延迟10秒更新可接受吗？
```

**幂等性设计**：

```
问题：网络故障导致请求重发 → 重复执行？

方案1：幂等设计
  ├── 用唯一请求ID
  ├── 服务端记录已处理ID
  └── 重复请求 → 返回之前的结果

方案2：补偿机制
  ├── 允许重复执行
  ├── 设计可逆操作（取消报名）
  └── 后台定时补偿
```

### 3.5 分布式图计算

**核心挑战**：图算法进入分布式环境后的新形态。

**图算法分布式化对比**：

| 算法 | 单机实现 | 分布式挑战 | 分布式方案 |
|------|---------|-----------|-----------|
| **BFS/DFS** | 队列/栈驱动 | 邻居跨节点 | Pregel消息传递 |
| **最短路径** | Dijkstra贪心 | 优先队列跨节点 | Bellman-Ford迭代 |
| **连通分量** | 标记传播 | 标签跨节点交换 | Label Propagation |
| **PageRank** | 矩阵迭代 | 稀疏矩阵分片 | 分布式迭代+聚合 |
| **社区发现** | 局部扩展 | 边分布不均 | 图分片+局部计算 |

**Pregel模型**（Google的分布式图计算框架）：

```
核心思想：BSP（Bulk Synchronous Parallel）模型

每一轮（Superstep）：
  ├── 1. 每个节点处理本地消息
  ├── 2. 发送消息给邻居（跨节点）
  ├── 3. 全局同步Barrier
  └── 4. 进入下一轮

示例：分布式PageRank
  初始化：每个节点PR=1/N
  Superstep 1-N：
    ├── 收到邻居的PR值
    ├── 计算新PR = 0.15 + 0.85 * sum(邻居PR/邻居出度)
    ├── 发送新PR给所有出边邻居
    └── 判断收敛（PR变化<阈值）则Vote to halt
```

**图分片策略**：

```
挑战：图分片如何最小化跨节点边？

策略1：哈希分片
  node_id % num_machines
  ├── 简单均匀
  └── 但跨节点边多（平均50%边跨节点）

策略2：Metis分片（基于图划分算法）
  ├── 最小化跨分区边
  ├── NP-hard问题，用启发式近似
  └── 效果好但预计算成本高

策略3：标签传播分片
  ├── 动态调整分片
  ├── 利用连通分量局部性
  └── 适合动态图
```

## 四、LLM时代视角

### 4.1 分布式训练与推理

**LLM分布式训练的核心问题**：

```
挑战：训练千亿参数模型
  ├── 单机内存不够
  ├── 单机计算太慢
  └── 需要分布式训练

方案1：数据并行
  ├── 每台机器有完整模型副本
  ├── 分批处理不同数据
  ├── 每步同步梯度（AllReduce）
  └── 通信成本：梯度同步

方案2：模型并行
  ├── 模型切分到多台机器
  ├── 每台机器只有部分参数
  ├── 每步传递中间激活
  └── 通信成本：激活传递

方案3：混合并行
  ├── 数据并行 + 模型并行
  ├── 流水线并行（Pipeline）
  └── 通信成本复杂化
```

**最新研究（2025-2026）**：

1. **分布式训练通信优化**：
   - Gradient Compression（梯度压缩）
   - Sparse Communication（稀疏通信）
   - Local Updates（本地多步更新再同步）

2. **分布式推理**：
   - Model Splitting（模型分割推理）
   - Speculative Decoding（推测解码）
   - Edge-Cloud Hybrid（边缘-云端混合）

### 5.3 War Story设计建议

**War Story 1：日志分析系统的教训**

```
场景：分析1TB日志数据

错误方案（LLM给出的）：
  "把所有日志传到中心机器，排序后统计"

问题：
  ├── 网络传输：1TB数据 → 传输成本巨大
  ├── 内存溢出：排序需要2TB内存
  └── 单点故障：中心机器失败 → 全部重来

正确方案：
  ├── Map阶段：每台机器统计本地日志
  ├── Shuffle阶段：按时间/类型聚合
  ├── Reduce阶段：聚合统计结果
  └── 容错：某节点失败 → 只重跑该节点

教训：通信成本和容错比计算复杂度更重要
```

**War Story 2：热点导致的系统崩溃**

```
场景：社交网络API

问题：
  ├── 某名人发帖 → 百万粉丝同时点赞
  ├── 哈希分片 → 该用户所在机器被压垮
  └── 整个系统不可用

解决方案：
  ├── 热点识别：监控节点访问频率
  ├── 热点分裂：复制热点节点到多台机器
  ├── 读写分离：写主节点，读复制节点
  └── 动态扩容：热点自动分裂

教训：分片策略需要考虑热点
```

**War Story 3：一致性的代价**

```
场景：电商秒杀系统

方案对比：
  ├── 强一致：每次操作同步 → 性能差，成功率高
  └── 最终一致：本地操作 → 性能好，可能超额

设计决策：
  ├── 库存扣减：强一致（不能超卖）
  ├── 订单创建：最终一致（可以延迟）
  └── 统计展示：最终一致（可接受延迟）

教训：根据业务语义选择一致性级别
```
