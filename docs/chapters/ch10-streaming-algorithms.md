# 第10章 从存储到流式：内存改变算法

## 本章核心问题

前面的很多算法默认你可以把全部输入放进内存，然后反复访问。流式场景改变了这个前提：

```text
数据一条一条到来。
数据总量可能大到无法完整保存。
很多记录只能看一遍。
答案可以近似，但误差必须可解释。
```

第10章讨论当内存不够、数据不断到来时，算法如何用摘要、采样和近似结构工作。

---

## 10.1 从“保存全部”到“保存足够信息”

假设你要统计直播间实时在线用户数。如果每个用户 id 都保存下来，当然可以得到精确答案。但如果平台规模巨大，内存成本会迅速上升。

流式算法的关键不是“丢弃数据”，而是回答：

> 为了完成这个问题，我真正需要保留哪些信息？

有些任务只需要一个计数器，有些任务需要一个固定大小的候选集合，有些任务需要概率型摘要。

---

## 10.2 一遍扫描与在线更新

最简单的流式算法是在线维护状态：

```python
count = 0

def consume(event):
    global count
    if event["type"] == "question":
        count += 1
```

这个例子太简单，但模式很重要：

```text
新记录到来 -> 用当前记录更新状态 -> 丢弃原始记录
```

后面很多高级算法都遵循这个模式，只是状态不再是一个整数，而是哈希桶、样本集合、候选项或概率摘要。

---

## 10.3 Top-k、重频项与 Misra-Gries

“找出出现最多的 k 个元素”是流式场景常见问题。精确统计需要保存所有不同元素的计数；如果不同元素非常多，空间会很大。

Misra-Gries 的直觉是维护少量候选人：

```text
如果新元素已经是候选人，就加一票。
如果候选席位未满，就加入候选。
如果席位已满，就所有候选一起减一票。
票数归零的候选被移除。
```

这不是魔法，它用受控的空间换取“不会漏掉真正重频项”的保证。它也提醒我们：近似算法不是随便猜，而是要说清误差边界。

---

## 10.4 Bloom Filter：允许误报，不允许漏报

Bloom Filter 适合回答：

```text
这个元素是否可能出现过？
```

它的特点是：

- 如果回答“不在”，一定不在。
- 如果回答“可能在”，可能误报。
- 空间远小于保存完整集合。

可以把它类比成门口的多重印章检查。没盖齐某些印章的人一定没来过；盖齐的人可能来过，也可能只是碰巧盖到了同样的组合。

---

## 10.5 采样：从全部数据中保留代表

当你无法保存全部数据，但希望之后还能做分析，采样是一种自然选择。

水塘采样解决的是：

```text
数据流长度未知，只能看一遍，如何让每个元素最终被选中的概率相同？
```

这类算法的价值在于：它不仅给出程序，还给出概率保证。学生在这里会重新遇到第1章的正确性问题，只不过正确性变成了概率意义上的正确。

---

## 10.6 Agent 时代的流式提示词

当 agent 给出“把所有数据读入列表再处理”的代码时，你要能追问：

```text
请改成流式版本。
输入可能无法完整放入内存，只能逐条读取。
请说明状态大小、每条记录更新时间、是否精确、如果近似则给出误差含义。
```

流式算法特别适合训练读者区分“样例能跑”和“规模能跑”。在 agent 时代，这个区分常常决定系统是否能上线。
