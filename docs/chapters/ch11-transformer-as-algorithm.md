# 第11章 Transformer：注意力作为算法

## 本章核心问题

这本教材不是机器学习教材，但如果它面向 agent 时代，就不能完全绕开 Transformer。我们不把 Transformer 当成神秘模型，而是把它拆成算法问题：

```text
如何表示 token？
如何让一个位置检索另一个位置的信息？
注意力的计算成本为什么高？
KV-cache 为什么能加速生成？
长上下文为什么会带来算法和系统压力？
```

第11章的目标不是训练模型，而是让学生能用算法语言理解现代 LLM 的核心计算。

---

## 11.1 Token、向量与表示

文本进入模型后，不再是字符或词语，而是一串 token。每个 token 会被映射为向量。

可以把向量理解成一种可计算的“语义坐标”。算法不直接比较汉字或英文单词，而是比较向量之间的关系。这个转换让语言问题进入线性代数和搜索问题的世界。

---

## 11.2 注意力：一次带权检索

自注意力可以先粗略理解为：

```text
当前位置提出一个查询 query。
所有位置提供 key 和 value。
query 与 key 的相似度决定从每个 value 取多少信息。
```

这和第2章的接口/缓存、第3章的查找、第4章的图传播都有关系。不同点在于，注意力不是返回一个确定条目，而是对许多位置做加权组合。

---

## 11.3 为什么注意力昂贵

如果序列长度是 `n`，每个位置都要和其他位置计算关系，朴素注意力的关系数量接近：

```text
n × n
```

这就是长上下文昂贵的核心原因之一。上下文变长，不只是“输入多一点”，而是可能让成对关系数量快速增长。

生活类比：如果一个班有 30 个学生，让每个人了解其他每个人的意见，大约是 900 组关系；如果有 300 个学生，就是 90,000 组关系。规模增长 10 倍，关系数量增长 100 倍。

---

## 11.4 KV-cache：生成时的缓存

生成文本时，模型一次产生一个新 token。如果每次都从头计算整段前文，会重复大量工作。

KV-cache 的直觉是：

```text
前文 token 的 key/value 已经算过，就保存下来。
下一步生成时，只计算新 token，并复用旧的 key/value。
```

这正好回到第2章的缓存思想：缓存节省时间，但要付出空间；上下文越长，缓存越大；多轮对话越长，系统压力越明显。

---

## 11.5 RAG、检索与注意力的边界

长上下文不是解决知识问题的唯一方式。RAG 先从外部知识库检索相关材料，再把材料放入上下文，让模型生成回答。

这会带来新的算法问题：

- 文档如何切分？
- 向量索引如何召回？
- 相似度高是否等于真正相关？
- 检索到的材料如何去重、排序和压缩？

这也是为什么本教材前面强调“知识库不是中心索引”。更好的知识组织不是只堆文档，而是让概念、案例、术语和人物能被组合、复用和审查。

---

## 11.6 Agent 时代的模型审查

读者不需要从零实现 Transformer，但应该能审查 agent 对 LLM 系统的说法：

```text
请解释这个方案的上下文长度、注意力成本、KV-cache 空间、检索召回和失败模式。
不要只说“使用大模型”，要说明模型在这个任务中承担哪类算法角色：分类、检索、排序、生成、规划还是验证。
```

这是本章和全书的连接点：算法知识让你不再只消费模型输出，而能判断模型系统的结构是否合理。

## 练习方向

1. 用表格解释 query、key、value 三者各自扮演的角色。
2. 计算序列长度从 1,000 增加到 10,000 时，朴素成对关系数量如何变化。
3. 审查一个“把所有资料塞进 prompt”的方案，指出上下文成本、检索质量和缓存空间问题。
