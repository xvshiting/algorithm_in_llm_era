# 第12章 算法设计方法论：设计算法的算法

## 本章核心叙事

终章不是“传统方法论 + LLM 应用”，而是把方法论本身看成一个算法：一个设计算法的算法。

六问诊断法可以直接翻译为 agent 的规划循环。把这六问编码为 Skill，就是本课程最后的综合项目。

---

## 12.1 六问诊断法

面对一个陌生问题时，不要立刻问“用什么算法”。先按下面六问推进：

1. **我真的理解问题吗？** 定义输入、输出、约束和小例子。
2. **有没有简单正确的算法？** 先写暴力或启发式基线，再分析它慢在哪里。
3. **我的问题在已知目录中吗？** 检查是否接近排序、查找、图、DP、贪心、搜索或 NP 完全问题。
4. **特殊情况是否可解？** 固定一个参数、缩小规模、删掉一个约束，再逐步推广。
5. **哪种范式最像？** 在分治、数据结构、图建模、动态规划、贪心、随机化、近似之间比较。
6. **还是想不出怎么办？** 回到问题定义，构造反例，请教专家或让 agent 生成候选并逐一审查。

正确回答“能不能这样做”不是只说“不能”，而是：

```text
不能，因为这个选择破坏了某个条件、漏掉了某类情况，或成本在某个规模下不可接受。
```

---

## 12.2 从六问到 agent 规划循环

六问法的每一步都可以编码为 agent 的决策节点：

| 六问 | Agent 对应动作 |
|------|----------------|
| 理解问题 | 解析输入输出、约束和目标 |
| 简单算法 | 生成基线、构造小样例、自测 |
| 查目录 | 检索知识库、匹配已知问题 |
| 特殊情况 | 简化问题、局部求解、再推广 |
| 范式匹配 | 比较算法范式和复杂度 |
| 循环修正 | 反例测试、专家调用、重新建模 |

这也是本教材知识库要服务的目标：不是把资料堆成中心索引，而是让 agent 能沿着概念、术语、人物、问题类型和失败案例做多路径联想。

---

## 12.3 试飞员心态

面对设计问题时，很多人会冻结：题目看起来陌生，于是直接等待灵感。

更专业的做法是跑清单。就像试飞员遇到异常时先执行检查序列，算法设计也应该先执行一套稳定的问题序列。你不必一开始就知道答案，但你应该知道下一步该检查什么。

Agent 也有类似问题。它不会表现出紧张，但会生成自信而错误的输出。试飞员心态要求我们让 agent 走清单，而不是直接跳到最终代码。

---

## 12.4 从简单到正确，从正确到高效

算法设计经常有三个层级：

```text
能解释问题的简单算法
能覆盖边界情况的正确算法
能承受目标规模的高效算法
```

不要一开始就追求复杂技巧。一个慢但正确的基线，可以帮助你构造测试、发现反例、理解瓶颈。优化不是把代码写得更花，而是有证据地移除瓶颈。

这条线贯穿全书：

- 排序前先问是否真的需要完整排序。
- 图算法前先问节点和边是什么。
- 动态规划前先写出状态和转移。
- 贪心前先寻找交换论证或反例。
- NP 完全问题前先判断是精确求解、近似、启发式还是限制规模。

---

## 12.5 策略先于战术

战术问题是：

```text
用邻接表还是邻接矩阵？
heapq 的 tuple 里要不要放计数器？
memo 的 key 怎么写？
```

策略问题是：

```text
这能建模为图问题吗？
这个子问题是否有重叠？
局部选择是否能推出全局最优？
输入规模是否迫使我们接受近似？
```

没有策略，战术会变成局部修补。Agent 很容易在战术层面给出大量代码，但策略判断仍然需要读者用算法知识审查。

## 12.6 终章项目：把六问法编码为 Skill

终章项目要求你设计一个 agent skill。输入是一个算法问题描述，输出是一份结构化诊断报告：

```text
问题理解：
输入输出：
规模与约束：
简单基线：
候选范式：
反例与边界情况：
复杂度比较：
推荐方案：
仍需人工确认：
```

这不是让 agent 替代算法学习，而是反过来：用算法学习来约束 agent 的推理流程。

---

## 练习方向

1. 用六问法分析一个陌生问题，记录每一步的推理。
2. 对一个 NP 完全问题，分别给出暴力解、贪心启发式和近似思路。
3. 让 LLM 对同一问题走六问法，评价每步质量。
4. 构造一个简单正确但慢的解，再逐步优化。
5. 完成终章项目：把六问诊断法编码为一个 agent skill。
